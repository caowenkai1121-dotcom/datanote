# 数据治理 & 主数据 模块 — 专业功能评估(Domain+Architect 视角)

> 视角: DAMA-DMBOK 11 知识域 + DCMM 八能力域 + 真实企业治理/MDM 落地流程。判据: 是否服务真实业务闭环、是否产生可执行动作、是否避免重复。三分类: **无用(砍)** / **需补全(残缺闭环)** / **需新增(专业缺口)**。

---

## 一、数据治理模块

### A. 无用 / 过度建设(砍或合并)
1. **大量纯可视化报表**(R117-118 已砍部分): 治理总览大屏/体检报告/五维雷达/趋势折线/90天热力、工单分析环图、联动地图 govmap。判据: 只展示计数、不驱动动作、与子模块重复。**剩余待砍**: 质量覆盖矩阵热力、安全态势分布环图、数据消费热度/排行/僵尸指标可视化、数据标准落地概览、各模块顶部概览磁贴中纯展示项。
2. **治理健康分(5维总分)的"分数剧场"**: 总分+雷达是 vanity metric, 真实价值在其驱动的【工单】与【DCMM自评】。建议: 健康分降权为一个入口数字, 不再围绕它堆图。
3. **数据消费(指标驾驶舱)与独立『指标管理』模块职责重叠**: 取值/新鲜度/僵尸指标本属指标管理。治理里再做一遍=重复。建议: 治理只保留"指标作为数据资产被消费的血缘/质量"视角, 驾驶舱归并到指标管理。

### B. 需补全(功能在、闭环残缺)
1. **质量失败→工单**: 已接(QualityService 自动建 issue) ✓。**但** 工单【解决】后无【复检】回链(关闭工单未触发对应规则复跑验证)。补: 工单 VERIFIED 时自动复跑来源规则确认修复。
2. **落标稽核→工单**: 落标违规未自动转工单(只出明细)。补: 稽核违规一键/自动转整改工单。
3. **分类分级→安全脱敏**: 打标(security_level)后, 脱敏策略需手工再配。补: 按密级自动建议/生成脱敏策略(高密级列默认脱敏)。
4. **数据 owner/责任人贯通**: 资产有 owner 字段但无"指派 owner / owner 看板 / owner 收敛其负责资产的待办"。补: 资产责任人指派 + 按 owner 的治理待办。
5. **质量规则模板库**: 现为手工填规则。补: 内置规则模板(空值率/唯一性/值域/正则/参照完整性/及时性/记录数波动), 选模板+表即生成。

### C. 需新增(专业缺口)
1. **数据访问申请/审批(data access request)**: 用户对敏感数据/数据集发起访问申请→数据owner审批→授时限权限。当前只有静态 RBAC + 脱敏策略, 缺"按需申请"闭环。这是金融/政企治理刚需。
2. **变更管理(schema change detection)**: 采集时对比上次 schema, 检出表/字段增删改→生成变更事件→按血缘通知下游 owner。当前采集是覆盖式, 无变更感知。
3. **数据质量评分卡(per 表/per 域)**: 现只有全局质量分。新增每张表/每域的质量评分(规则覆盖率×通过率), 驱动 owner 针对性整改。
4. **数据资产申请上架/认证(certified data)**: 资产经质量+标准+owner确认后标记"已认证", 消费侧优先用认证资产。

## 二、主数据(MDM)模块

### A. 无用 / 过度建设
1. **主数据总览**(R118 已砍报表→KPI hub) ✓。
2. **数据管家**与【黄金记录草稿筛选 + 去重】功能重叠: 是聚合待办跳转页, 价值有限但成本低, 暂留(后续可并入黄金记录的"待办"标签)。

### B. 需补全(功能在、闭环残缺) —— 最关键
1. **【本轮已修】变更审批 approve 不应用变更**: 原 review() 只改 status, 不动黄金记录→审批=摆设。**已补**: 批准即应用(create 新建 / update 合并 payloadJson / delete 软删 inactive), 应用失败则保持 pending 报错。这是 MDM 变更治理的命门。
2. **发布订阅 publish 是模拟记账, 非真推送**: MdmPublishService.fanOut 只写 publish_log status=success, 从不 HTTP 调 endpoint。补: 对订阅 endpoint 真实 HTTP POST(黄金记录 JSON), 落真实成功/失败+重试。否则"分发"是假的。
3. **直接改黄金记录绕过审批**: 黄金记录模块 save/publish/delete 直接落库, 与审批并行两条路。补: 高敏感实体的黄金记录写【强制走审批】(实体加 require_approval 开关), 审批成为唯一写口。

### C. 需新增(专业缺口) —— MDM 核心方法论缺失
1. **源记录接入→清洗→匹配→存活→黄金记录 的整合管线**: 当前黄金记录靠【手工录入】, 这不是 MDM 本质。MDM 本质=多源记录消费+合并成单一可信版本。新增: 【从源表导入】(选数据源+表+字段映射到实体属性)→批量生成候选记录→自动 match(已有去重)→survivorship(已有规则)→生成/更新黄金记录, xref 记录源ID映射。这是 MDM 最大缺口, 让 dedup/survivorship/xref 真正串起来。
2. **数据质量分(per 黄金记录完整度)**: 现质量监控只判约束合规。新增每条黄金记录的完整度/可信度评分(必填齐全率+来源数+冲突数), 驱动数据管家优先治理低分记录。
3. **黄金记录历史版本/审计追溯**: 现 version 字段递增但无版本快照查看(改了什么、谁改的)。新增变更历史(diff)。

## 四、已落地
- **R119 · MDM 审批批准→真实应用变更**(P0): review() 批准时 applyChange — create 新建黄金记录 / update 合并 payloadJson 进 dataJson(version+1) / delete 软删(inactive); 应用先于改状态, 失败保持 pending 报错; 回填 goldenRecordId 便追溯。审批从"记状态"变为"真治理"。
- **R120 · MDM 发布订阅真实推送**(P0): MdmPublishService.fanOut 不再谎报 success——配了 endpoint 就用 HttpURLConnection(零依赖)真实 POST 黄金记录, 按响应码记真状态(2xx=success / 其余=failed+HTTP码 / 异常=failed+原因 / 无endpoint=skipped), 3s 超时防拖死, 去 @Transactional 避免长事务。验: 不可达→Connection refused/DNS失败, httbin→真实 HTTP 码。分发从"假记账"变为"真推送"。
- **R121 · MDM 源表导入整合管线**(P0 命脉): 新增「源数据接入」——MdmIngestService 从源库表【消费记录】→ 按映射组装属性 → 按(源系统,源ID)+业务主键【匹配】→ 命中【合并】(源值覆盖, version+1)否则【新建】黄金记录, 维护 XREF(源ID↔黄金记录ID), 幂等(重导更新同一记录)。批内按源ID去重 + 预载 xref/golden 内存匹配(免逐行查库提速)。黄金记录从"手工录入"转为"多源消费合并", 把 dedup/存活/xref 真正串起来。新增 mdm-ingest.js 前端(实体+源库表+字段映射+导入)。验: xh_dms.t_sales_order 100订单行→9个不同客户(customer_code去重), 幂等(重导 created=0/updated=9 无重复), 18s。
  - 剩余 P1: 落标违规→工单、工单关闭→规则复检; 按密级自动生成脱敏; 质量规则模板库。

## 二、四大重复 / 职责重叠（重构核心靶点）

> 注：以下为**真重复**（应合并），业务域已剔除技术域误判的"伪重复"（见第五章）。

### 2.1 调度 / 执行：5–6 套并存
- `LocalSchedulerController(/api/scheduler)` 实际调外部 **DolphinScheduler**（仅 2 处真实调用）。
- `SchedulerConfigController(/api/scheduler)` **同基础路径**，仅放 config 读写 → **路由冲突隐患**。
- `TaskSchedulerService`(650 行本地引擎) + `TaskExecutionService` + `TaskDependencyService`。
- `BackfillController` + `BaselineController`（补数/基线，又各一套执行编排）。
- `sync/` 包**自带** `SyncJobScheduler/SyncJobExecutor/DepReadyChecker`。
- **6 种上下线组合**（local/online × script/sync）实现高度重复。

### 2.2 同步引擎：2 套并存（须区分真伪）
- 老：`DnSyncTask` + `DataxService`（DataX），入口 `/api/datax`、`/sync-task`。
- 新：`DnSyncJob` + 自研 `sync/` 引擎（全量/增量/CDC、Debezium、StreamLoad、对账、坏行、schema 漂移）。
- ⚠️ **业务域纠偏**：二者对应**两个真实场景**（开发侧"一键采集" vs 运维侧"生产管道"），DataX 仍被 10+ 文件活跃引用。Product 主张"自研为唯一标准、DataX 体系废弃"；Domain 主张"分工保留"。**此为待决策分歧点**（见 `待我决策.md`）。

### 2.3 元数据 / 资产：3 套并存
- `MetadataController(/api/metadata)` = 数据地图，`DataMapService`(648 行 / 27 方法) **既直连库 SHOW TABLES 又读 `DnTableMeta`**（在线探查与离线目录混在一个 service）。
- `MetadataCenterController(/api/metadata-center)` = 目录 CRUD + crawl 采集。
- `AssetDetailController(/api/gov/asset)` = 治理视角资产详情/画像/术语。
- 三者都围绕 `DnTableMeta/DnColumnMeta`，profile/detail 端点重复；且 `dev.html` 的"表元数据编辑" workspace 进不去 → 真割裂。

### 2.4 概览 / 健康 / 打分：4+1 套并存
- 全局 `OverviewController` + `DashboardController` + 治理 `HealthController` + `HealthScoreService`。
- Project 模块**自带** `/{id}/overview` `/{id}/health` + `ProjectHealthScorer`（与全局重复打分）。
- 各处靠重复 SQL 聚合，无事件驱动。

### 2.5 其它真重复
- **告警两套**：脚本 `AlertConfigService` + 同步 `sync/AlertService`。
- **审计三套**：主库 `dn_audit_log` + 同步 `dn_sync_job_audit` + 标签 `dn_label_audit`。
- **用户偏好四张同构表**：表收藏 / 搜索历史 / 项目收藏 / 项目访问。
- **DDL 双路由** `/api/doris` + `/api/hive`（Hive 已弃用，纯历史债）。
- **认证分裂**：`AuthController`(login/status) 与 `RbacController`(me/用户角色权限) 各管一段。
- **配置三处**：`SystemConfig` / `AlertConfig` / `SchedulerConfig`。
- **MDM 过度拆分**：11 个 Controller（多个仅 1–2 端点）。

## 四、业务闭环断点（重构最该投入处 —— Domain 独立判定）

理想闭环：`接入 → 加工 → 资产 → 治理 → 消费 → 反馈`。逐段体检：

| 环节 | 状态 | 断点 |
|---|---|---|
| 接入 / 加工 / 资产 | ✅ 通 | 数据源 + 同步 + 脚本调度 + 元数据血缘主干扎实 |
| **接入 → 资产** | 🔴 **断点1** | 采集后元数据**不自动登记**，开发建了 Doris 表数据地图未必搜得到 |
| **质量 → 工单** | 🔴 **断点2** | 质量规则失败**不自动生成治理工单**，治理"发现→督办"核心闭环断裂 |
| **治理 → 加工（脱敏）** | 🔴 **断点3** | 脱敏只在 DDL 改写，脚本执行 `/api/doris/execute` **可绕过** → 安全有洞 |
| **消费** | 🔴 **断点4** | 指标定义后**无消费出口**，缺消费层，治理成果无法变现 |
| 反馈 | ⚠️ 弱 | 健康分黑盒、工单未被质量自动驱动 |

> 主干（接入→加工→资产）通且有价值；**真正断的是治理自动化三接点 + 缺失的消费层**。而 MDM/项目协作/CDC 等"藤蔓"恰不在主闭环上 —— 印证其为伪需求。

## 六、团队对"推翻重建"的裁决

- 🔴 **【Domain】行使一票否决权，否决"字面推翻重建"**：主干健康有真实用户，72 轮 + 生产验证的稳定性重写一年换不来；病根是**产品定义问题（伪需求）+ 闭环断点**，非技术债，推翻会把伪需求原样再造。
- 🟢 **【Architect】**：工程上"推翻"≠"停机重写"，应**绞杀式（Strangler Fig）分轮迁移**，以目标架构为锚，每步可运行可部署可回滚。
- 🟢 **【Product】**：不是"推翻所有重建"，而是"保留验证过的闭环 + 砍重复/孤立/低值 + 重组边界 + 前端模块化"。

**PM-Lead 拍板**（岔路口机制，记入 `待我决策.md` 供业主醒后复核）：
> 采纳**绞杀式渐进重构**路线。它达成业主"彻底重构为清晰、联动、无冗余目标架构"的目标（该推翻的伪需求照砍），同时满足业主全局红线（小步可运行、外科式、禁整文件重写）。唯一不采纳的是"删光已验证主干从零重写"——业主自身全局红线亦禁止此。

## 七、风险与守门红线

- **红线1**：绝不停机重写；分轮绞杀，任一轮独立可部署可回滚。
- **红线2**：历史 SQL 01–65 只增不改，整合走 66+ 幂等迁移 + 回滚说明。
- **红线3**：旧端点先转发后删除，删除前全局检索确认零调用。

- **安全债**：`deploy_jar.py` 明文密码且被 git 跟踪、生产 IP/密钥硬编码 → 见 `待我决策.md`。

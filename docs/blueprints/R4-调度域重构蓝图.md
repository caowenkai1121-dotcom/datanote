# 调度域结构重构蓝图(架构师)

## 推荐
调度/执行域 第1轮：抽取 ScheduleLifecycleService 统一8种上下线 + 将 /api/scheduler 按职责拆为 Lifecycle/Config/Execution/Monitor 四个门面控制器（旧端点全部以 @Deprecated 薄转发保留，前端零改动）。目标：消灭脚本与同步任务上下线的逐字重复，让调度域职责单一、像新系统，且失败可见、无数据迁移、可单轮绞杀完成。

### 分层与职责单一化

```
表现层（4个门面 Controller，按职责单一拆分，前缀仍为 /api/scheduler 兼容前端）
├── ScheduleLifecycleController   职责：上下线（脚本/同步 × 本地/远程，8端点）
│      仅做参数校验 + 委托 ScheduleLifecycleService
├── SchedulerConfigController     职责：调度参数 CRUD（已存在，保持不变，4端点）
├── ScheduleExecutionController   职责：引擎控制 + 执行控制（start-daily/stop/retry/
│      rerun/stop-task/resume/pause-downstream/backfill/batch-retry/cron-preview）
└── ScheduleMonitorController     职责：监控查询（today/run-log/logs/log-detail/task-runs）

应用/领域层（新增一个服务，吸收上下线全部业务）
└── ScheduleLifecycleService（新建）
       核心方法（统一脚本与同步两类的上下线）：
       - Map onlineRemote(Long id, ScheduleTargetType type)   // DS 远程上线
       - void  offlineRemote(Long id, ScheduleTargetType type) // DS 远程下线
       - void  onlineLocal(Long id, ScheduleTargetType type)   // 本地上线（含版本快照+依赖刷新）
       - void  offlineLocal(Long id, ScheduleTargetType type)  // 本地下线
       内部私有：
       - applyDsResult(...)        // 统一回写 dsProjectCode/WorkflowCode/TaskCode/ScheduleId+ONLINE
       - setScheduleStatus(...)    // 统一置 ONLINE/OFFLINE（消4处重复）
       - buildSyncShellScript(DnSyncTask) // 同步 DataX shell 生成（从 Controller 搬入）
       - createOnlineVersion(DnScript)    // 版本快照（从 Controller 搬入）
       依赖：DolphinService / DnScriptMapper / DnSyncTaskMapper /
             DnScriptVersionMapper / TaskDependencyService

枚举（新建，消除 if(script)/if(sync) 分叉）
└── ScheduleTargetType { SCRIPT, SYNC }
       封装两类目标在 mapper、字段读取上的差异（策略式分派）
```

### 模块边界
- 上下线"业务"全部下沉到 ScheduleLifecycleService，Controller 退化为薄校验+转发。
- DS 交互仍只经 DolphinService（不动）。
- 调度引擎 TaskSchedulerService / 执行 TaskExecutionService / 依赖 TaskDependencyService 本轮不动（仅被新 Service 调用），降低爆炸半径。
- 旧 LocalSchedulerController 全部端点保留为 @Deprecated 薄转发到新4门面对应方法，HTTP 行为逐字不变。

### "像新系统"的判定
- /api/scheduler 路由按 lifecycle/config/execution/monitor 四类语义聚合，单一职责；
- 8种上下线归一为 4 个统一方法 + 一个 type 枚举，重复代码清零；
- 后续可一次性删除旧转发层而无需碰前端。

注：本轮**不**做"统一本地调度引擎(P0竞态)""统一运行表(双轨制)"——那两项含业务逻辑变更与数据迁移，留待后续轮，本轮严守"行为不变"。

## 数据迁移
无。本轮为纯代码结构重构：不新增/修改任何表、不动 dn_scheduler_run / dn_task_execution / dn_backfill_task / dn_baseline_task 字段。运行表双轨制统一与本地调度引擎合并（含数据迁移）属后续轮，本轮严守"行为不变、零迁移"以保证可单轮安全完成。若硬性要求占位，可预留空文件 sql/66_noop_placeholder.sql（仅含注释，幂等）但实际无 DDL/DML——建议本轮直接不建。

## 风险与回滚
风险点：
1) 路由冲突——4 个新门面 + 旧 LocalSchedulerController 共享 @RequestMapping(\"/api/scheduler\")，若新旧定义了同一 method+path 会 Spring 启动报 Ambiguous mapping。缓解：旧类端点改为转发但若与新门面同路径会冲突——策略二选一：(a) 旧类端点物理删除、由新门面承接原路径（前端零改动，因路径字面量不变）；(b) 若需 @Deprecated 显式留痕，则新门面用 /lifecycle/** 新路径、旧类保留原路径转发。推荐(a)：路径字面量不变即前端零改动，最干净；@Deprecated 加在旧 Service 包装或注释中。本蓝图按(a)为主、(b)为备。
2) 私有方法迁移遗漏——createOnlineVersion 依赖 DnScriptVersionMapper，迁 Service 时须同步注入，否则 NPE。缓解：单测覆盖本地上线路径。
3) @Value 注入位置——datax.home/job-dir 必须随 shell 生成逻辑迁到 Service，旧 Controller 若残留引用会启动失败。缓解：编译期即可暴露。
4) DataX shell 文本细微差异（空格/换行）会导致同步任务行为变化。缓解：迁移时整段复制不重写，并以字符串断言锁定。

回滚策略：本轮无数据变更，回滚=纯代码 revert。git 单提交（或单 PR）承载本轮全部改动，出问题 git revert 该提交即恢复，无需任何数据修复。部署侧无 SQL 需回退。建议先在 caowen 环境部署冒烟 24 端点可达 + E2E 绿后再合主干。

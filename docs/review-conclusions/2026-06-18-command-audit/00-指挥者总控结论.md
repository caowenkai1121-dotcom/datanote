# 00-指挥者总控结论

## 总判断

【PM-Lead】本项目已经具备中台雏形：数据源、数据地图、治理、主数据、项目、调度、AI、审计等能力都已经落地，不是单纯堆菜单。但当前最大问题不是功能数量，而是“入口太多、边界不够硬、体验不够人性化、部分能力仍像半成品”。如果直接继续堆功能，会让系统越来越重、用户越来越迷糊。

【PM-Lead】后续 Claude 的角色是实施者，不负责重新拍板方向。Claude 必须按本目录任务顺序修复，不要部署，不要写入远程服务器信息，不要把凭据写进任何文件。

## 本轮最高优先级

1. 【Architect】先封堵跨用户与旁路风险：公共 WebSocket topic、旧 DataX/DDL 入口、SQL SSE executionId、AI 样例数据读取。
2. 【Dev-A】再收紧安全与权限：CSRF、AI 审批归属、Shell 执行、SQL 权限粒度、DataX 临时明文文件清理。
3. 【Design】再修用户可感知的大体验问题：启动鉴权失败放行、旧弹层、无权限入口说明、小屏顶栏、登录开放模式。
4. 【Domain】清理无意义入口：凡是没有完整业务动作、没有完成定义、只显示“即将完善”的功能，必须删除、隐藏或补成闭环。
5. 【Product】把低频/隐藏模块转成明确跳转或归并说明，不能让用户进入无侧栏高亮、无上下文的“幽灵页面”。

## 强制禁止

- 禁止部署。本轮只是写审查结论，后续 Claude 修改完成后再由用户决定部署时机。
- 禁止记录服务器 IP、密码、token、私钥、2FA、提现 key 等任何凭据。
- 禁止一次性重写 `workspace.html`、`project.js` 或大面积格式化。
- 禁止新增第三方依赖，除非用户另行确认。
- 禁止把“权限不足”简单做成隐藏入口了事；用户需要知道缺什么权限、找谁处理。
- 禁止保留“即将完善”这类无交付定义入口。
- 禁止把 P0/P1 问题推给文档解释；必须改代码并有验证。

## 实施节奏

【PM-Lead】建议 Claude 分三批执行：

1. P0：安全边界与真实入口，优先封堵跨用户、旁路、半成品入口。
2. P1：易用性和权限一致性，提升人性化与可解释性。
3. P2：架构收敛和维护性，降低长期复杂度。

每批只改一组强相关文件，完成后运行对应测试。不要在同一批里同时改前端壳、权限模型、DataX、AI、文档，这样很难回归。

## 放行标准

【QA】Claude 修复后必须同时满足：

- 关键路径可运行，无控制台明显报错。
- 权限失败必须 fail-closed，不能静默进入工作台。
- 无权限页面或入口必须能解释缺失权限点。
- 高危执行类能力必须绑定用户、任务、会话或审批上下文。
- UI 空态、加载态、错误态、禁用态齐全。
- P0/P1 任务有测试或可复现实验步骤。
- `mvn -q test`、必要的 `node --check` 通过。

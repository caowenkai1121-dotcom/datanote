# 04-架构审查

## 总体结论

【Architect】系统方向不是问题，问题在于架构边界还没有完全硬化：旧入口和新领域服务并存，数据读取策略分散，前端工作台壳层过重，文档滞后于真实实现，配置契约偏开发态。

【Architect】Claude 后续不要“大重构”，应按边界逐步收敛：先封统一策略，再迁调用方；先抽小公共能力，再拆大壳层。

## P0：建立统一数据读取策略

当前表级 ACL、脱敏、行过滤、limit、审计分散在多个调用方。

证据：

- `src/main/java/com/datanote/platform/iam/DataAclService.java:23` 数据 ACL 服务独立存在。
- `src/main/java/com/datanote/domain/datasource/DatasourceExploreService.java:208` 预览链路读取数据。
- `src/main/java/com/datanote/domain/datasource/DatasourceExploreService.java:244` 预览链路已有 fail-closed 脱敏策略说明。
- `src/main/java/com/datanote/domain/governance/AssetDetailService.java:179` 资产详情样例读取。
- `src/main/java/com/datanote/platform/ai/agent/tool/impl/TableDataTool.java:54` AI 工具读取样例数据。
- `src/main/java/com/datanote/platform/ai/agent/analysis/AnalysisQueryService.java:42` AI 分析链路声明 ACL 和脱敏。

问题：

- 有的入口走脱敏，有的入口直接读样例。
- 上层调用方需要“自觉补 ACL”，这是不可维护的。

Claude 修复要求：

- 新增或收敛为 `DataReadPolicyService` / `TableAccessPolicy`。
- 提供 `filterTables / canReadRows / canSeeMetadata / rewriteForMasking / auditRead` 等最小接口。
- `AssetDetailService.sampleRows`、AI `table_data`、元数据预览逐步接入同一入口。

## P0：旧入口和新领域服务必须统一

证据：

- `src/main/java/com/datanote/domain/integration/DataxController.java:93` 旧 DataX 运行入口。
- `src/main/java/com/datanote/domain/integration/service/SyncJobService.java:87` 新同步域已有数据源访问校验。
- `src/main/java/com/datanote/domain/integration/HiveDdlController.java:70` 起包含 Hive DDL 能力。

问题：

- 同步域已经有任务、数据源、权限、执行记录，但旧入口仍能绕过一部分规则。

Claude 修复要求：

- 不要一次性删除旧接口。
- 先把旧接口内部调用统一 guard。
- 再在文档标注旧接口为兼容层。
- 最后由产品判断是否隐藏旧入口。

## P1：前端工作台壳层过重

证据：

- `src/main/resources/static/workspace.html:19` 加载大量业务模块。
- `src/main/resources/static/workspace.html:3496` 全局 API 薄封装。
- `src/main/resources/static/workspace.html:7796` 路由切换同时处理权限、抽屉、锁释放、DOM 显示。
- `src/main/resources/static/workspace.html:17635` 到 `17653` WebSocket 订阅也在壳层。

问题：

- 路由、权限、API、WebSocket、编辑锁、页面显示耦在一个 HTML，任何小改都容易牵连。

Claude 修复要求：

- 外科式拆出 `api-client.js`、`router.js`、`ws-client.js`。
- 每次只迁一个横切能力，保留兼容全局函数。
- 页面模块只保留 `render/init` 和本模块状态。

## P1：API 契约和 DTO 需要收敛

证据：

- `src/main/java/com/datanote/domain/metadata/MetadataController.java:40` 使用 `Map<String, Object>` 请求。
- `src/main/java/com/datanote/domain/metadata/MetadataController.java:155` `aiSearch` 使用 Map。
- `src/main/java/com/datanote/common/SubjectController.java:42` 树接口返回 `List<Map<String, Object>>`。
- `src/main/java/com/datanote/common/SubjectService.java:30`、`87` 也用 Map 拼树。
- `src/main/java/com/datanote/domain/consumption/ConsumptionController.java:58` 起大量 Map 响应。

问题：

- Map 适合探索期，但现在接口多、前端多、AI 工具多，契约不稳定会增加回归风险。

Claude 修复要求：

- 不要全仓一次性替换。
- 先从高频和安全敏感接口开始：metadata、datamodel、schema diff、project tags。
- 新增最小 DTO，保留响应字段不破坏前端。
- 同步补契约测试。

## P1：配置管理偏开发态

证据：

- `src/main/resources/application.yml:18` 开启 `allow-circular-references`。
- `src/main/resources/application-example.yml:13` 示例也开启循环依赖。
- `src/main/java/com/datanote/domain/integration/connector/ConnectionManager.java:41` extraParams 直接拼 JDBC URL。
- `src/main/java/com/datanote/domain/integration/connector/ConnectionManager.java:50`、`63`、`76` 拼接不同数据库参数。

问题：

- 循环依赖开关说明服务边界仍有缠绕。
- JDBC extraParams 如果无白名单，可能形成连接参数注入或绕过安全默认项。

Claude 修复要求：

- 补配置契约：Env > DB system config > yml 的优先级。
- 新增 `JdbcExtraParamsSanitizer` 白名单。
- 循环依赖先记录依赖链，再分步拆，不要盲关配置。

Claude 修复要求：

## P2：AI、向量、图、Redis 的架构定位

【Architect】AI/向量/图/Redis 不是无意义功能，但要明确职责：

- AI：辅助搜索、分析、生成治理动作，但不能绕过权限。
- 向量：资产/术语/字段语义检索，需要 ACL 过滤。
- 图：血缘和影响分析，需要增量镜像。
- Redis：缓存和体验锁，不承担强一致。

证据：

- `src/main/java/com/datanote/platform/cache/CacheService.java:52` 使用 `redis.keys(prefix + "*")`。
- `src/main/java/com/datanote/platform/ai/vector/VectorIndexService.java:34` 向量索引服务。
- `src/main/java/com/datanote/platform/ai/graph/GraphMirrorService.java:23` 图镜像服务。

Claude 修复要求：

- Redis 前缀清理改 SCAN 或维护 key 索引集合。
- 关键写入靠数据库 version 乐观锁，不靠 Redis 锁保证强一致。
- 图和向量保留全量校准，但新增变更事件驱动的增量更新路线。

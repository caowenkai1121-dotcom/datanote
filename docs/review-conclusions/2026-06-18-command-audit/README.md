# 2026-06-18 指挥者审查结论

本目录用于存放本轮多角色深度审查结论。当前阶段只做审查与指挥，不改业务代码，不部署，不记录任何远程连接信息或凭据。

## 角色分工

- 【PM-Lead】统一调度、归并结论、定义 Claude 实施顺序。
- 【Design】审查 UI 人性化、易用性、设计系统一致性、空态/错误态/权限态。
- 【Dev-A】审查后端、安全、权限、执行链路、数据泄露风险。
- 【Architect】审查模块边界、旧新入口并存、数据读取策略、前端壳层、配置与文档。
- 【Domain】从真实生产场景判断功能是否成立，识别无意义功能和断裂流程。
- 【Product】把问题转成可实施需求，明确保留、合并、隐藏、删除和补齐路径。
- 【QA】定义后续 Claude 改完后的验收门槛。

## 文件说明

- `00-指挥者总控结论.md`：总判断、优先级、实施原则、禁止事项。
- `01-UI与易用性审查.md`：前端 UI/UX、人性化、可访问性和无意义入口审查。
- `02-业务产品审查.md`：业务闭环、产品价值、模块去留、流程断点。
- `03-后端安全审查.md`：权限、安全、执行链路、数据泄露与运维风险。
- `04-架构审查.md`：架构边界、统一策略、前端壳层、配置治理。
- `05-Claude实施任务清单.md`：给 Claude 的分批修复工单。
- `06-QA验收清单.md`：修复后的回归、手测和放行标准。

## 给 Claude 的使用方式

1. 先读 `00-指挥者总控结论.md`，确认禁止部署、禁止写凭据、禁止一次性重写大文件。
2. 再按 `05-Claude实施任务清单.md` 从 P0 到 P2 小步修复。
3. 每完成一组任务，按 `06-QA验收清单.md` 运行对应验证。
4. UI 相关改动必须同步遵守 `docs/设计规范.md`。
5. 不要把本目录当成“建议池”，P0/P1 是必须闭环的实施项。

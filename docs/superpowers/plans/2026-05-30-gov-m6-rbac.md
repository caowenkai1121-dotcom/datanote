# 数据治理 M6：多用户 + RBAC 底座 实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans。步骤用 `- [ ]` 勾选跟踪。

**Goal:** 建立 RBAC 底座（`dn_user`/`dn_role`/`dn_user_role`/`dn_role_perm`），把单 admin 内存认证升级为「MySQL 用户体系 + 配置 admin 兜底」双数据源，提供用户/角色/分配 CRUD 与 `/api/rbac/me` 端点，治理中心「数据安全」模块出用户/角色管理界面。**安全第一：默认仍可登录、密码 BCrypt 哈希、最小改动、不锁死任何人。**

**Architecture:**
- `RbacService` 负责用户/角色 CRUD、给用户分配角色、查用户权限集（聚合其所有角色的 perm_code）、纯函数 `hasPermission(Set<String> perms, String need)`（`*` 通配或精确匹配）。密码经 `BCryptPasswordEncoder` 哈希后入库。
- `SecurityConfig` 接入 `DbUserDetailsService`（从 `dn_user` 加载用户名/BCrypt 密码/角色），作为 `DaoAuthenticationProvider`；**同时保留** 原 `inMemoryAuthentication` 的配置 admin 作为兜底 provider。`dn_user` 表不存在 / 查询异常时，`DbUserDetailsService` 吞异常抛 `UsernameNotFoundException`，由内存 provider 接管，绝不 500。
- 认证总开关语义 **保持不变**：`datanote.auth.password` 为空 → `isEnabled()=false` → 全放行（开源默认开发态）；非空 → Session 认证启用，两个 provider 并存。

**Tech Stack:** Java 8 / Spring Boot 2.7 / spring-security-crypto BCrypt / MyBatis-Plus；JUnit 5；vanilla JS。

**迁移耦合提示：** 部署前先应用 `sql/36_rbac.sql`（幂等，含预置 ADMIN 角色 + admin 用户 + `*` 全权限）。预置 admin 默认口令见下方「安全注意点」。

**参照：** `docs/superpowers/specs/2026-05-30-data-governance-design.md` §7 M6。

## 安全设计核心（红线）

1. **默认可登录**：`isEnabled()=false`（密码未配）时全放行，行为与现状完全一致，开发态零影响。
2. **不锁死**：启用认证后，DB 用户 + 配置 admin **双数据源**。即使 `dn_user` 未建表 / 为空，配置 admin（`datanote.auth.username/password`）仍能登录。
3. **不报 500**：`DbUserDetailsService` 对 SQL 异常（表不存在等）降级为「用户未找到」，链路平滑回退到内存 provider。
4. **密码不明文**：入库一律 `BCryptPasswordEncoder` 哈希；预置 admin 用预生成 BCrypt 串。
5. **最小改动**：既有 `.java` 仅改 `SecurityConfig.java` 与 `AuthController.java`；其余全部新建。

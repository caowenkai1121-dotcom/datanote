# 第 2 轮 · 治理闭环加固：质量恢复 → 自动关单

> 路线图 R2。主题：补全 R1 闭环的另一半 —— 质量规则**恢复达标**时，自动关闭仍处 OPEN 的该规则工单，形成"发现问题 → 督办整改 → 恢复关单"完整双向闭环。

## 本轮范围调整说明（PM-Lead 决策）
原 R2 计划含三件：①b 恢复关单、② 脱敏统一编排、③ 分类→标签回写。调研测绘后**重排**：
- **② 脱敏（preview）拆出专门轮**：`/api/doris/execute` 与 stream-execute 现已套脱敏（断点3 的执行面其实已在旧轮修复）。真实残留绕过是 `DataMapService.preview`（`SELECT * ...`）。给它脱敏需正确处理 **`SELECT *` 列展开** + 复用现私有于 Controller 的 **用户上下文/绕过判定** + fail-closed；ETL 写入路径（TaskExecution/Backfill）**绝不能脱敏**（会把脱敏值写进目标表毁数据）。该轮需前置设计，不本轮硬塞。
- **③ 分类→标签回写拆出专门轮**：`dn_table_meta.tags` 字段在服务层几乎无读写、格式未确立，贸然写入有破坏前端渲染风险，需先定 tags 格式（CSV/JSON）。
- 本轮只做 **①b 恢复关单**：100% 安全加法、全在已吃透代码区、E2E 可验，质量优先于堆功能。

## 本轮目标
- 质量规则执行结果为 `success`（恢复达标）时，若该规则存在仍为 **OPEN**（未被人工流转）的质量工单，自动置 CLOSED 并追加关单说明。
- 人工已介入（FIXING/RESOLVED/VERIFIED）的工单**不自动关**，尊重人工整改流程。
- 与 R1 复用同一去重键 `qrule:{ruleId}` 与同一收口点，零表变更。

## 各角色产出
- **【Domain】** 闭环不能只开不关：规则修好后工单该自动消解，否则工单越积越多失真。但"人工正在处理中(FIXING)的单子"不能被系统抢着关 —— 只自动关无人动过的 OPEN 单。判定标准：规则先失败出 OPEN 单 → 修规则使其达标 → 复跑后该单自动 CLOSED。
- **【Architect】** 复用 R1 收口点 `QualityService.executeRule → IssueService.raiseQualityIssue`：失败分支建单（R1），成功分支调 `closeQualityIssueOnRecovery`（R2）。仍全程 try/catch 兜底，不影响质量执行。关单策略纯函数 `shouldAutoCloseOnRecovery(status)` 可单测。
- **【Product】** 无新端点、无新页面：自动关的工单在既有工单列表自然消失（status=CLOSED 默认列表按 updated_at 倒序仍可见已关单）。描述追加"[自动关单] …"留痕。
- **【Dev-A】** `IssueService`：成功分支 → `closeQualityIssueOnRecovery`（仅关 OPEN 单 + 追加说明 + 落库）；新增纯函数 `shouldAutoCloseOnRecovery`。
- **【QA】** 见下。

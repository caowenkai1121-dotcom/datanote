# 第 3 轮 · 治理闭环接点③ 分类确认 → 表级敏感标签回写

> 路线图原 R4（执行顺序前移：tags 格式已核实为安全的 CSV，可先做此安全加法；preview 脱敏因需更重设计顺延到后续专门轮）。
> 主题：列级分类确认后，自动在 `dn_table_meta.tags` 维护表级"含敏感字段"标签，让数据地图/资产视图能一眼识别敏感表，打通"分类 → 资产标签"联动（对应 `00-现状调研.md` 孤立点：confirm 后不回写表标签）。

## 本轮目标
- 列分类确认（`/api/gov/classification/confirm`）后，按"该表是否仍有敏感列（`sensitive_type` 非空）"**幂等增/删**表级标签 `含敏感字段`。
- 自愈：降级（清空 sensitive_type）后若全表再无敏感列，自动移除该标签。
- 兼容既有 tags 格式（逗号/中文逗号分隔 CSV，前端 `split(/[,，]/)` 渲染、后端 `like` 过滤），其余标签保序保留。
- 兜底不抛：标签回写失败不影响分类主流程（confirm 为 @Transactional，回写异常已 try/catch）。

## 各角色产出
- **【Domain】** 分类做完，资产侧要能"看见"哪些表含敏感数据，否则分类成果不流转到数据地图。判定：确认某列敏感→该表出现"含敏感字段"标签；把该列降级→标签消失（前提全表无其它敏感列）。沿用既有"敏感=sensitive_type 非空"约定，不另造口径。
- **【Architect】** 复用 `ClassificationService` 既有 `tableMetaMapper`/`columnMetaMapper`，零新依赖零表变更。敏感判定与 `sensitiveHeatmap` 同口径（`sensitive_type` 非空），避免双标准。标签增删纯函数 `applySensitiveTag(tags, hasSensitive)` 可单测；表态由 `syncTableSensitiveTag` 按全表实时计数驱动（自愈、幂等）。
- **【Product】** 不新增端点/页面：标签写入既有 `dn_table_meta.tags`，数据地图/资产列表既有渲染即可显示，`metadata-center/tables?tag=含敏感字段` 可直接筛敏感表。
- **【Dev-A】** `ClassificationService`：confirm 末尾调 `syncTableSensitiveTag(tableMetaId)`（try/catch 兜底）；新增纯函数 `applySensitiveTag` + 常量 `SENSITIVE_TAG`。
- **【QA】** 见下。

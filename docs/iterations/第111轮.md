# 第111轮 —— 深挖智能化+提速:单轮并行多工具(减往返)+ 对抗审查加固

> 业主: 在智能化和提速上深度挖掘。

## 一、核心结构性提速 —— 单轮并行多工具
**现状**: 主循环每轮只执行【第1个】tool_call(`toolJsons.get(0)`)。查3张表=3次 LLM 往返(每次 pro~60s/flash~20s), 或走 delegate_task(子代理各自多轮 LLM, 实测189-248s)。
**改进**: LLM 一条回复里给【多个相互独立的只读 tool_call】时, 后端【并行执行、一轮拿齐结果】:
- AiAgentService 主循环新增并行批分支: 解析全部 tool_call → 资格校验(只读 Guardrail.PASS / 非 PARALLEL_EXCLUDE{ask_user,delegate_task,cron_job} / 参数合法 / 未死循环) → 全部合格则用 aiChildExecutor 池并行 invoke(复用子代理池, ≤MAX_PARALLEL_TOOLS=4, 各60s墙钟) → 主线程串行写 trace/step/previews(避 st.seq 竞态) → continue。任一不合格回退原单工具串行路径。
- PromptBuilder 协议#2: 允许多个独立只读 tool_call 同回复并行, 写/有依赖/ask_user/delegate 仍一次一个。
- 比 delegate_task 更轻(无子代理多轮 LLM 开销)。
**实测**: "查3表字段结构"→agent 一轮发3个 asset_detail 并行→**57s**(vs delegate 189-248s, ~3-4x); 写步结构验证 think 仅挂批首步。

## 二、对抗审查工作流(ultracode)加固
并行批改动落在核心循环且贴近写审批区, 起 14-agent 对抗审查工作流(并发/安全/正确性三镜 find→逐条 verify 验真), 查出 **3 个真 bug 全修**:
1. **批内重复未去重**(seenCalls 只查跨轮持久 map, 同回复重复同一调用两个都过)→ 加 `batchSeen` 本地 Set 构建期去重。
2. **预算欠扣**(批接纳不看 budget.remaining, 余量<批量时多执行步绕过 MAX_PRODUCTIVE_STEPS=8)→ 构建期 `cap = min(MAX_PARALLEL_TOOLS, budget.remaining()+1)` 限批量, 保证补扣不溢。
3. **RUNNING 标记泄漏**(批处理中 writeStep 抛异常则 marker 不删→幽灵"运行中"步)→ 批处理体包 try/finally, finally 删 marker。
- 安全验证: 写操作(create_project)仍 wait_approval 走审批门(并行批排除写工具, 未绕过); 复验无残留 RUNNING 步。

- (注)flash 偶尔仍会叠加 delegate_task 致变慢(LLM 规划方差); 协议已引导"能并行就并行", 后续可继续收敛。

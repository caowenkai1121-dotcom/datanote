# 第121轮 —— MDM 命脉补全(P0): 源表导入→匹配→存活→黄金记录 整合管线

> 续路线图 P0。专业判断: MDM 本质 = 多源记录【消费/匹配/合并】成单一可信版本, 而非手工录入。此前黄金记录靠手填, dedup/存活/xref 成孤岛。本轮补上整合管线把它们串起来。

## 一、后端: MdmIngestService(整合管线)
从源库表消费记录 → 黄金记录, 幂等:
1. **读源行**: AssetDetailService 重构出 read(db,table,n) 共享; 新增 readRows(上限2000, 与 sampleRows 同库源/数仓路由)供批量导入。
2. **组装**: 按 mapping(源列→属性编码)取值, 空值不写。
3. **必填校验**: 缺必填属性的行跳过(不污染黄金记录), 计 skipped + 错因。
4. **批内去重**: 按(可选 sourceIdColumn 否则业务主键)对源行去重, 同源ID后值覆盖——避免同客户多订单行把版本翻倍。
5. **匹配**: 预载该源系统 XREF(sourceId→goldenId)+该实体未停用黄金记录(id/bizKey→记录)到内存; 逐去重源ID先按 XREF 命中, 退化按业务主键命中。
6. **合并/新建**: 命中→源值覆盖映射字段合并进 dataJson + version+1 + updateById; 未命中→新建黄金记录(active/draft)。均维护 XREF(源ID↔黄金记录ID)。
7. **提速**: 内存匹配 + 只写变化(不再逐行查库), 解决远端逐行查询超时。

## 二、入口 + 前端
- MdmIngestController: GET /api/mdm/ingest/columns(读源列供配映射) + POST /api/mdm/ingest/from-table。
- mdm-ingest.js「源数据接入」(MDM 侧栏新模块, 接 黄金记录 后): 选目标实体→填源库.表→加载源列→字段映射(源列下拉选属性, 同名自动匹配, 标注关键/唯一/必填)→选源ID列(幂等键)/源系统/导入即生效/最多行→开始导入→结果统计(读取/新建/更新/跳过 + 错因 + 跳查黄金记录)。

## 三、QA(本地连远程库 E2E + pymysql + Playwright)
- 导入 xh_dms.t_sales_order(customer_code→cust_id, customer_name→cust_name, 源ID列=customer_code, 100行) → **100订单行去重为 9 个不同客户黄金记录**(dn_mdm_golden_record + dn_mdm_xref 各对应), skipped=6(行缺必填如实跳过)。
- **幂等**: 重复导入 created=0 / updated=9(命中已有 XREF→更新同一记录, 无重复), 证整合管线幂等。
- **提速**: 批内去重+预载内存匹配后, 100 行 18s(优化前逐行查库 >60s 超时)。
- 前端: 源数据接入 入 MDM 侧栏(12→13), 表单/实体下拉/字段映射渲染正常。

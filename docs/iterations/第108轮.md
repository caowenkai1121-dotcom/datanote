# 第108轮 —— 修3个体验bug:运行卡死假象 / 刷新丢历史 / 模型档位复位

> 业主反馈: ①提问"查看下 xh_dms 库中销售数据相关的表"一直卡在"天工司辰运行中·正在理解你的问题…" ②刷新页面历史记录丢失 ③选了快速模式刷新后又恢复默认。

## 一、Bug1: 运行"卡在正在理解"假象(根因+修)
**根因**: agent 调长耗时工具(尤其 delegate_task 并行子代理, 实测阻塞 ~140s)时, 后端在工具 invoke【之后】才落库 SKILL_CALL 步; invoke 期间会话无任何新步 → 前端轮询取不到进度 → 副标题冻结在"正在理解" → 看似卡死(实际在跑)。
**修**:
- 后端 AiAgentService: 工具 invoke【前】插一条 `RUNNING` 进度标记(seq 复用不自增, 调用后即删), 让前端轮询实时读到"当前正在跑哪个工具"。`writeRunningMarker()`。
- 前端 ai-agent.js 轮询: 读 RUNNING 标记显示"· 正在{人性化动作}（第N步 · 已Xs）"; 即便尚无工具步也显示"· 正在理解你的问题…（已Xs）"——【始终带计时】, 用户一眼知道在跑不是卡死。
- E2E 验: 强制 delegate 场景, 前端 31s→170s 全程显示 `RUNNING=delegate_task`(不再冻结); 自然 query 显示 asset_detail→read_tool_result 实时序列。

## 二、提速(回答太久)
- **子代理默认走⚡快速档**: ChildAgentRunner 新增 `datanote.ai.child-model`(默认 deepseek-v4-flash)。子代理做的是机械取数(table_profile/asset_detail 列字段), 快速档足够, 综合质量由父代理保证。实测同场景 248s→189s。
- **⚡快速档贯穿到子代理**: 子代理在新线程不继承父 ThreadLocal 的 MODEL_OVERRIDE; runBatch 显式捕获父档位下传(`getModelOverride()`), 用户选⚡快速时父+子全快速, 端到端提速。
- **精炼 delegate 纪律(防过度委派)**: 协议第10条改为——简单多对象取数直接主循环依次调工具(更快, 委派每子代理自身有多轮 LLM 开销); 仅当【每个子任务需独立多步分析/推理】才用 delegate_task 并行。修正 R107 过度鼓励委派致简单查询反而慢。验: "查看销售表"不再委派, 改用 asset_detail+read_tool_result 直查。

## 三、Bug2: 刷新丢历史(修)
**根因**: 对话仅存在前端 DOM, 刷新即丢(会话/步骤其实都在 DB)。
**修**: ai-agent.js
- sessionId 持久化到 localStorage(`aiSessionId`); runRequest/renderTurn 落地。
- build() 末调 `restoreHistory()`: 拉上次会话全部步骤, 按"用户气泡→执行过程折叠→终答气泡"只读重建(不重跑)。
- 加 `✚ 新会话` 按钮(清历史+重置 sessionId)。

## 四、Bug3: 模型档位刷新复位(修)
**根因**: selectedModel 仅内存变量。
**修**: modelPicker 读/写 localStorage(`aiModel`), 构建时回填 select 选中项, onchange 持久化。

# 第120轮 —— 修 MDM 发布订阅"假分发"(P0): 模拟记账 → 真实 HTTP 推送

> 续路线图 P0。业主铁律: 功能要可用、有意义。

## 根因
MdmPublishService.fanOut 对命中订阅【直接写 publish_log status=success + "已推送至…"】, 【从不调用 endpoint】——"分发"是假记账, 订阅方永远收不到黄金记录。

## 修复
真实推送(零依赖 JDK HttpURLConnection):
- 配了 endpoint → HTTP POST 黄金记录载荷(goldenRecordId/entityId/bizKey/changeType/status/version/data/publishedAt) → 按响应码记【真状态】: 2xx=success / 其余=failed+HTTP码 / 异常=failed+原因。
- 未配 endpoint → status=skipped(不再谎报 success)。
- 3s 连接/读超时(防订阅端不通拖死发布); 去掉 @Transactional(避免 HTTP 在长事务内); 单订阅推送失败不影响其它(best-effort 扇出)。
- buildPayload 把 dataJson 解析为对象嵌入; message 记目标系统+endpoint+结果, 截断 480。

## QA(本地连远程库 E2E)
建实体1的订阅(不可达/可达 httpbin)+ 发布黄金记录触发 update 扇出, 查 publish_log:
- http://dw.internal/... → failed「推送异常: dw.internal」(DNS 不通)
- http://127.0.0.1:9/hook → failed「推送异常: Connection refused」
- https://httpbin.org/post → failed「HTTP 503 推送失败」(真实 HTTP 往返拿到响应码)
→ 证明确在真发 HTTP, 按真实结果记录, 不再恒 success。验毕删除测试订阅。

## 验收门禁自检
- [x] 发布订阅真实 HTTP POST + 真状态(2xx/失败/异常/skipped), 不再假记账

- [x] E2E 验证真实推送结果(不可达/异常/HTTP码)

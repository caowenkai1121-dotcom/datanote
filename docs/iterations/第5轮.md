# 第 5 轮 · 元数据/资产域 结构重构（DataMapService god-service 拆分）

> 第 2 个真·结构重构轮（架构师 #2 ROI 靶点）。消除"一个 service 既连库实时探查、又读离线目录"的核心债。
> **行为不变 / 端点不变 / 前端零改动**：仅拆服务、重路由控制器内部调用。

## 问题（架构师定调）
`DataMapService`（**648 行 / 27 方法**）把两类完全不同的职责揉在一起：
- **在线探查**：直连 Doris 跑 `SHOW DATABASES/TABLES`、`DESCRIBE`、`SELECT *`、`SHOW PARTITIONS`、`information_schema` —— 实时、依赖 BE 存活。
- **离线目录/社交**：读 `DnTableMeta`/收藏/评论/搜索历史 —— 持久态、与库存活无关。
二者共用一个类、一份 `HiveConfig` 连接，边界模糊、难测难维护。

## 目标架构（职责单一化）
```
DatasourceExploreService（新建，唯一直连库出口）  ← 仅依赖 HiveConfig
  getHiveDatabases/getHiveTables/getHiveColumns(+getDorisColumns/getColumnsByDescribe)
  getAllTablesSummary / preview / profile / generateDdlAndSelect
  getPartitions(+mapDorisPartitionRow) / getDorisTableInfo
DataMapService（瘦身为离线资产目录）  ← 依赖 mappers + AiAssistService + DatasourceExploreService
  searchTables / aiSearch / 搜索历史(3) / 收藏(3) / getPopularTables / 评论(3) / getTableDetail
  —— 凡需实时库数据均委托 exploreService，**自身不再持有任何 DB 连接**
MetadataController（/api/metadata，端点一字不变）
  在线端点(databases/tables/columns/all-tables/preview/profile/ddl/partitions) → exploreService
  离线端点(search/ai-search/history/favorites/popular/comments/table-detail) → dataMapService
```

## 团队验证（沿用 R4 蓝图工作流结论）

- **【Product】**：`/api/metadata/*` 22 端点全被 workspace.html 使用，**路径与响应契约一字不变**（仅控制器内部改调哪个 service），前端零改动、零功能丢失。

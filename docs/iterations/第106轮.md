# 第106轮 —— agent 运行反馈优化: 提交后有进度 + 动画思考 + 结果优先

> 业主反馈(图 tp/2.png): ①ask_user 卡片【提交后无任何进度反馈】, 似 agent 卡死/没干活, 误导用户。②思考过程要简略 + 有动画 + 让用户知道在运行; 最后只展示结果数据, 不铺思考过程的数据。

## 一、根因
- ask_user 卡片 finish() 直接 `DN.post(/answer)` → 期间(续跑一整轮, 数十秒)无任何指示, 卡片只变灰, 用户以为卡死。
- 旧 send() 运行中把【完整工具卡(含原始结果JSON)】实时铺出 + renderTurn 末尾又把全部工具卡+think+plan 全量渲染 = 思考过程数据淹没结果。

## 二、统一"带动画进度执行器" runRequest(send 与 ask_user 提交共用)
- **动画 thinking**: 注入一次 CSS keyframes(旋转 spinner + 脉冲点); 运行气泡 = 旋转 spinner + "天工司辰运行中" + 简略副标题。
- **运行中简略进度**: 每 1.5s 轮询 GET /session/{sid}, 副标题更新为"· 正在{工具友好名}（第N步）"(TOOL_LABEL 映射, 只显动作不暴露参数/结果)。
- **结果优先渲染**: 完成移除动画 → renderTurn 只出【最终答复】+ 折叠的"🔧 执行了 N 步 · 点击查看过程"(默认收起; 点开才渲工具卡+计划)。不再默认铺思考过程/计划/think 数据。
- **send() 与 ask_user finish() 都改走 runRequest** → 提交确认后立即有动画+进度(修复①), 不再似卡死。

## 三、QA(Playwright 真机)
- **运行态截图**: 用户气泡下出现「🔄 天工司辰运行中 · 正在分派子任务（第1步）」动画进度条, 发送按钮"思考中…"。
- **结果态截图**: 仅展示最终治理报告(P0-P3 建议表 + 一句话总结), 冗余工具数据折叠进"执行了 N 步"(wait_for"执行了"命中)。
- ask_user 提交复用 runRequest(同动画机制), 提交后即见运行进度(修复无反馈)。

# 第101轮 —— hermes 复刻 W6 续: 模型热切(/model)+ Smart审批安全取舍

> Ultracode 续推。W6 余项: 模型热切(/model)——同 provider 下按会话切模型档位(快速省时 vs 高质量)。Smart 审批预筛因触及安全关键审批路径, 经评估【刻意不做】(见下)。

## 一、模型热切 /model
- **AiAssistService**: 新增 ThreadLocal `MODEL_OVERRIDE` + 静态 set/clear; chat() 用覆盖模型优先于快照(同 provider, 仅换 model 档位)。
- **AiAgentController.chat**: 请求体可带 `model`, 请求边界 set/clear(finally 清, 防泄漏到其它请求/异步线程)。
- **前端 ai-agent.js**: 头部模型档位下拉(默认模型 / ⚡快速[deepseek-v4-flash] / 🎯高质量[deepseek-v4-pro]); send() 随 chat 上送 model。
- 简单查询用快档省时省钱, 复杂分析用强档 —— 按工序重要度选模型(务实成本控制)。

## 二、QA
- **覆盖生效**: 传 model="no-such-model-xyz" → status=blocked, 报错"supported models: deepseek-v4-pro 或 deepseek-v4-flash"(证明 override 真送达 provider; 且发现 flash 快档)。
- **无泄漏**: 不传 model → status=done(默认模型, ThreadLocal 请求边界 clear 无跨请求泄漏)。
- **前端**: Playwright 真机截图 —— 头部「默认模型 ▾」下拉 + 「⏰定时」「🧠经验」「审批」按钮全渲染, 融入设计系统, 39 工具。

## 三、Smart 审批预筛 —— 刻意不做(安全取舍)
蓝图列了"辅助 LLM 预筛写审批(approve/deny/escalate, HIGH 仍人工)"。评估后【不实施】, 理由:
- ApprovalGate 是写操作的【安全咽喉点】, 刚经对抗审查加固(R99)。引入 LLM 自动批准决策 = 在最关键安全路径上加一个可被提示注入/模型误判操纵的决策点, 与"写操作必经人工审批 + fail-closed"红线张力大。
- 审批疲劳当前非真实痛点(写工具少, 多为按需)。收益小、风险高。
- 故保持现状: 所有写操作一律人工审批(HIGH/MEDIUM 同), 不引入 LLM 自动放行。如未来确有疲劳, 再以"仅 escalate 提示、绝不自动 approve"的保守形态评估。

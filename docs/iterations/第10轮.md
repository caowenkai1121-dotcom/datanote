# 第 10 轮 · 新增「数据消费层」域 — 填补断点④治理成果无消费出口（产品+架构师+业务专家深挖）

> 业主明确"重构 = 改造 + **新增功能**"。本轮深挖团队早先识别的最大真实断点 **④治理成果无消费出口**（指标定义后无人消费、无报表/看板/API、数据资产无法变现），**新增 `domain.consumption` 数据消费层**（契合刚建成的领域驱动结构）。
> 6-agent 工作流（产品设计功能群 / 架构师设计落地 / 业务专家定 MVP+红线）。

## 业务专家裁决（有条件通过 + 砍伪需求）
- **MVP（本轮做）**：① 指标执行引擎 + 指标值快照表 → 让"定义"产出"值"（缺它消费层全是空壳）；② 指标取值/历史查询 API；③ 指标消费看板（值/趋势/对比，留 R11 前端）；④ 结果导出（突破预览 500 行，含审计）；⑤ 消费审计透视（谁/何时/消费什么 → 僵尸指标识别）。
- **红线（已遵守）**：消费不绕治理 —— 复用 `HiveService.executeSQL` 执行引擎、不另造查询通道；**只消费 status=1 已发布指标**（未发布/僵尸指标拒绝取值）；所有计算/查询/导出写 `dn_consumption_log` 审计。
- **砍伪需求（本轮不做）**：API 密钥市场 / OAuth2 外部接入 / 订阅推送(MDM pubsub 伪发布) / 计费 / GraphQL/OpenAPI / 物化视图 —— "内部数据平台无外部消费方"，不符真实场景，待真有外部需求再立项。

## 新增内容（domain.consumption）
- **表（sql/66_consumption.sql，幂等）**：`dn_metric_value`（指标值时序快照）、`dn_consumption_log`（消费审计流水）。
- **模型/Mapper**：DnMetricValue/DnConsumptionLog + 2 Mapper（已加入 @MapperScan）。
- **MetricValueService（指标值引擎）**：复用 `HiveService.executeSQL` 按 `DnMetric.calc_formula` 计算→落快照；latest/history/batchLatest/freshness/zombies/overview/logConsumption + 纯函数(parseMetricValue/firstCell/isStale/ageHours)。
- **ConsumptionController（/api/consumption，11 端点 = ≥10 新功能点）**：
  1. POST `/metric/{id}/calc` 计算单指标值
  2. POST `/metric/calc-all` 批量计算全部启用指标
  3. GET `/metric/{id}/value` 最新值
  4. GET `/metric/{id}/history` 值时序
  5. POST `/metric/values` 按 code 批量取值
  6. GET `/metric/freshness` 新鲜度
  7. GET `/metric/zombies` 僵尸指标
  8. GET `/overview` 消费层概览
  9. GET `/log/list` 消费审计流水
  10. GET `/log/heat` 消费热度 Top
  11. GET `/metric/{id}/export?format=csv|json` 导出(防注入+审计)

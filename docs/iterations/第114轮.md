# 第114轮 —— 深挖智能化:工具自动补全可推导参数(别问能自查的)

> 业主反馈图 tp/1.png: "xh_dms.t_sales_order 新建ODS任务,把这个表抽到ODS层" → agent 却反问用户"源数据源ID(datasourceId)是多少?"。这类本可自动识别的,不够智能,应 agent 自动补全。

## 一、根因
create_ods_table / create_sync_job 把 datasourceId/sourceDsId 设为【必填】。agent 不知道就 ask_user 反问用户。但该 ID 完全可由【库.表】从 dn_table_meta(已采集元数据, 每表带 datasourceId)查到——不该让用户填本可省去的空。

## 二、改进:工具自动补全 + 协议自动补全纪律
1. **AssetDetailService.resolveDatasourceId(db, table)**: 按库表解析所属数据源ID(精确表优先, 退化同库任一表)。
2. **CreateOdsTableTool**: datasourceId 改【可不填】; invoke 时若未给则 `resolveDatasourceId(db,table)` 自动补全; 实在查不到才返回 need_datasource(提示采集/确认, 而非直接甩给用户)。描述/schema 同步: "直接带 db、table 调用即可"。
3. **CreateSyncJobTool**: sourceDsId 改【可不填】; 给 sourceDb 即 `resolveDatasourceId(sourceDb,null)` 自动识别源数据源。
4. **PromptBuilder 协议第16条 自动补全纪律(智能·别问能自查的)**: 用户给了【库.表】后, 工具所需【可推导参数】(datasourceId 自动识别/ODS 表名按规则生成等)一律自己补全或交工具自动识别, 直接带 db/table 调用, 绝不为能查到的信息去 ask_user; ask_user 只留给真正需用户拍板的决策。能自动完成就一步到位。

## 三、QA(本地连远程库 E2E, flash 档)
- 复现图场景 "xh_dms.t_sales_order 新建ODS任务" → agent 直接调 **create_ods_table args={db:xh_dms, table:t_sales_order}**(无 datasourceId, 没反问用户) → 工具自动补全 datasourceId → **status=wait_approval**(写操作进审批门, 25s)。对比图中 wait_input 反问 datasourceId, 已修。
- 自动补全用 dn_table_meta.datasourceId(R109 源库取数已证 xh_dms 各表该字段齐备), 审批通过后 invoke 即解析成功。

# 第 18 轮 · 指标输入质量联动（消费↔治理质量跨域，新功能）

> 接 R15（DnMetricRef 指标-资产血缘）+ R10-16 消费层。本轮打通 **消费域指标 ↔ 治理域数据质量**：指标驾驶舱可一键查看「该指标来源表的数据质量」，给指标**可信度信号**——输入数据不达标的指标，其值不可轻信。
> 红线遵守：纯**读聚合**，复用 `DnMetricRef`（R15）+ `dn_quality_rule` + `dn_quality_run`（治理域既有），不新建表、不改写路径、零迁移。

## 角色

- **【Domain】** 真实场景：指标消费者看到一个数字，却不知其底层表是否脏。指标可信度 = f(输入数据质量)。此前消费域与质量域**割裂**——看指标的人看不到输入质量。本轮接通：来源表有失败质量规则 → 指标标记「输入存在风险」。场景成立。
- **【Architect】** 联动方向：consumption → governance 单向（consumption 引 DnMetricRef/QualityRule/QualityRun mapper，反之不引），无环。聚合在 `MetricValueService`，controller 仅转发。无新表、无新依赖。
- **【Product】** `inputQuality(metricId)`：来源表(DnMetricRef 去重) → 每表启用质量规则(dn_quality_rule status=1) → 每规则最新执行(dn_quality_run by started_at) 的通过率/状态 → 整体信号 `HEALTHY / AT_RISK(有失败) / NO_RESULT(规则未跑) / NO_RULES`。
- **【Design】** 复用 `DN.drawer/pill`，指标行加「输入质量」操作 → 抽屉：顶部信号药丸 + 各来源表分组 + 规则行(严重度药丸 / 名称·维度 / 通过率药丸 红绿)。
- **【Dev-A】** 后端：`MetricValueService.inputQuality`（注入 `DnQualityRuleMapper`/`DnQualityRunMapper`）；`ConsumptionController` 加 `GET /metric/{id}/input-quality`。
- **【Dev-B】** 前端：`gov-consumption.js` opCell 加「输入质量」→ `inputQualityDrawer`。版本 q100r84→q100r85。
- **【QA】** 见下。

## 新增内容

- **后端（domain.consumption，零新表）**：`MetricValueService.inputQuality(metricId)` + `ConsumptionController.GET /api/consumption/metric/{id}/input-quality`。
- **前端**：`gov-consumption.js` 指标行「输入质量」操作 + `inputQualityDrawer`（信号药丸 + 来源表分组 + 规则通过率）。版本 **q100r85**。

## 一、新增 4 个写工具(全部走 R65 既有咽喉点: Guardrail→ApprovalGate→审计→invoke)
| 工具 | 风险 | 映射 UI 同款入口 | 建成在哪可见 |
|---|---|---|---|
| `create_sync_job` | MEDIUM | SyncJobService.save(含服务端校验:必填/模式/cron/tableConfig) | 数据同步 模块列表 |
| `create_ods_table` | **HIGH** | HiveService.generateDDL+executeDDL(读源表字段→Doris建表) | 数据地图 ODS 库 |
| `create_quality_rule` | MEDIUM | /quality/rule/save → ruleMapper.insert | 治理健康-质量规则 |
| `create_metric` | MEDIUM | /metric/save → metricMapper.insert | 指标管理 |

- 工具数 26→30(7 写 + 23 读)。零注册自动收集(@Autowired List), 无需改注册表。
- 身份字段(createdBy/owner)一律取 ctx.userName, 绝不取模型输出。
- `create_ods_table` 加标识符护栏(库/表名仅 [a-zA-Z0-9_]+, 杜绝 DDL 注入)+ syncMode 白名单(df/di); 源库口令用 CryptoUtil.decryptSafe 解密, 不落明文。
- `create_sync_job` 复用 SyncJobService.save 的服务端校验, 非法入参以 bad_arguments 返回(不裸抛)。

## 三、QA 打回→本轮闭环
- **缺陷**: create_quality_rule 首测 resume=done 但规则表空。根因: `dn_quality_rule.datasource_id` 列 NOT NULL 无默认值, 工具允许 null→DB 拒(`Field 'datasource_id' doesn't have a default value`)。UI 表单总传该值。
- **修复**: datasourceId 改必填(schema required:true + invoke 前置校验 bad_arguments)。重测 PASS(规则 id=9)。
- **旁证框架健壮性**: 该工具失败时, 框架未崩(resume 仍 done), agent 自行诊断出"缺 datasourceId + ods.t_order 不存在"并给出可读结论 —— 写工具失败优雅降级, 不污染会话。

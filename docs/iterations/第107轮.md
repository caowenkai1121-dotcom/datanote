# 第107轮 —— 修结果截断 + 多对象并行提速 + 进度词人性化

> 业主反馈(图 tp/1.jpg): agent 答 xh_dms 销售表时 资产探查状态多为"⚠部分获取/结果被折叠未完整解析/并行查询摘要未展示核心字段/因工具反复截断仅前5字段" —— 工具结果被截断, 字段看不全。另: 回答太久(多对象应并行子代理)、运行提示要人性化。

## 一、结果截断修复(核心)
根因: 结果上限过小, 表/字段清单被折叠致 agent 看不全 + 误判"截断/未获取"。放宽:
- TRACE_RESULT_CAP 1500 → **4000**(agent 在 trace 看到的单工具结果, 字段清单整张放下不折叠)。
- STORE_RESULT_CAP 8000 → **16000**(read_tool_result 可取更全)。
- ChildAgentRunner CHILD_RESULT_CAP 1200 → **3000**, 子小结上限 1500 → 3000, 子续跑/收尾 prompt 加"保留关键明细(表名/字段名/类型/数值)不要过度概括"——并行子代理汇总不再丢字段。
- 压缩阈值 12000 → **18000**(大结果会话不过早压缩)。
- PromptBuilder 提速纪律加"某步结果被折叠时用 read_tool_result(seq) 取全量, 不要据残缺结果下'截断/未获取'结论"。

## 二、多对象并行提速
PromptBuilder 协议第10条改"**提速纪律(重要)**: 查/析【多个对象】(多表字段/多血缘/多指标)时【优先】用 delegate_task 把每对象分派给一个子代理【并行】完成, 比逐个串行快很多, 不要一个个串行查"。

## 三、运行提示词人性化
前端 TOOL_LABEL 改自然短语(接"正在…"后): 检索相关的表 / 翻看字段明细 / 阅读你上传的文件 / 并行分派子任务 / 取回完整结果 等; runRequest 发起即显"· 正在理解你的问题…", 工具运行时"· 正在{动作}（第N步）"。简短告知用户在干嘛。

## 四、QA(本地连远程库 E2E)
- **截断修复+并行**: "逐表查 xh_dms t_sales_order/detail/his 字段结构" → 用 **delegate_task 并行**(3表并发), status=done, 答复 **2689 字含完整字段结构表**(序号/字段名/类型/键/非空/描述), **无"截断/折叠/未获取/未完整"残缺词**(对比图中"仅前5字段")。
- 进度: 运行态显人性化"正在…"提示(R106 动画 + 本轮自然措辞)。

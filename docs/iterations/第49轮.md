# 第49轮 —— 后端服务深度优化（第五批：质量/总览/元数据/采集/脚本/血缘）

## 本轮目标
延续后端深优主线，覆盖 R9 后**一直未深优**的 6 个核心服务（质量引擎、治理总览、元数据查询、元数据采集、脚本管理、血缘边）。逐函数审查，**防 NPE + 入口校验 + 异常归一**，行为语义不变。

## 各角色产出

### 【Architect】优化范围（6 服务，均自 R9 后未深优）
| 服务 | 优化点 | 红线 |
|---|---|---|
| QualityService | `failureAnalysis`：`selectList` 结果判空 + 遍历元素判空；`executeRule` 入参 `rule` 判空 `BusinessException` | SQL 注入校验/只读校验/规则执行语义不变 |
| OverviewService | `groupCount`：`selectMaps` 结果判空 + 行元素判空（原依赖外层 catch，改显式守卫） | 各子块容错聚合语义不变 |
| MetadataService | Oracle owner 解析 `conn.getMetaData().getUserName().toUpperCase()` 抽 `currentSchema()` 助手，**getUserName 为 null 时返空串防 NPE**（2 处复用） | 多源（MySQL/PG/SQLServer/Oracle）元数据查询语义不变 |
| MetadataCrawlerService | `crawlAll`：`selectList` 判空 + 元素/缺主键脏数据跳过；`crawlDatasource` 入参判空 + `IllegalArgumentException`→`BusinessException` 归一 | 增量 upsert（技术覆盖/业务空才填）语义不变 |
| ScriptService | `getTree`：4 个 `selectList` 统一兜底空列表；`buildTree` 四处遍历元素判空；`getById`/`save`/`updateBasicInfo`/`listVersions` 入参校验；`allWithContent`/`createScriptVersion`/`deleteFolderRecursively` 结果判空 | 树构建/版本管理/CRUD/递归删语义不变 |
| LineageEdgeService | `rebuildFromSyncJobs`/`bfs`/`graph` 三处 `selectList` 判空 + 边元素判空 | BFS 深度防环、子图 maxNodes、纯函数血缘构建语义不变 |

### 【QA】结论 —— ✅ 通过
- 编译：`mvn -o compile` 通过（新增 import：MetadataCrawler/ScriptService 的 `BusinessException`）。
- 全量回归：`mvn -o test` **518 run / 0 fail / 0 error / 1 skip（Docker testcontainers）** → **BUILD SUCCESS**。
- 行为等价核对：本批均为「结果/入参/元素」判空与异常类型归一，**无算法/查询/分支逻辑改动**；`currentSchema` 仅在 `getUserName()` 返回 null 时改变（原会 NPE，现返空串），属纯健壮性增强。

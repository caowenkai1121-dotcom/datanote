# 第122轮 —— 治理闭环联动(P1): 工单关闭/解决 → 规则复检(防假关闭)

> 业主铁律: 每改动深评业务价值 + 功能联动, 拒假功能/孤立功能。

## 深度评估(为何做这个)
- 业务价值: 质量工单若能被【人工随手关闭而不验证数据】, 则"工单已关"是假象——数据还坏。这正是"假功能"。
- 现状联动(已存在): 规则执行失败→自动建工单(objectRef=qrule:{id}); 规则再次执行恢复达标→自动关 OPEN 工单。
- 缺口(本轮补): 人工把质量工单流转到【已解决/关闭】时, 系统不复验数据 → 可假关闭。
- 联动设计: 工单(IssueService) ↔ 质量规则(QualityService) 双向打通——关单前【重跑规则】, 数据真达标才放行。

## 实现
IssueService.transition(): 当工单为 QUALITY 类型且目标态为 RESOLVED/CLOSED 时, reverifyQualityRule:
- 由 objectRef `qrule:{id}` 解析规则 → qualityService.executeRule(rule) 重跑;
- run=failed(数据仍低于阈值) → 抛"数据复检未通过: 通过率X% < 阈值Y%, 不能标记为已解决/关闭; 请先修复数据再操作" → 拦下流转(工单保持原态);
- run=success → 放行(真修复); run=error 或 规则已删/非规则工单 → 放行(不因数据源不通等基础设施问题困住用户)。
- 打破环: QualityService 依赖 IssueService(失败建单), 故 IssueService 反向用 @Lazy @Autowired 注入 QualityService。
- 前端无需改: 拦截抛 IllegalStateException → HealthController 转 R.fail(msg) → 前端既有 toast 显示原因。

## QA(本地连远程库 E2E)
- 建必失败规则(xh_dms.t_sales_order.delivery_type 非空, 阈值100) → 复跑 status=failed passRate=1.54% → 自动建工单(qrule:10, OPEN)。
- 试 OPEN→CLOSED → 被拦: code=-1 "数据复检未通过: ...通过率 1.54% < 阈值 100.00%, 不能标记为关闭; 请先修复数据再操作"; **工单仍 OPEN(假关闭被阻止)**。
- error/success/非规则工单 放行路径按设计(error 不困住用户)。验毕删测试规则+工单。

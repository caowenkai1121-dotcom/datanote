# 第90轮 —— 智能体更聪明:全服务点亮 + 知识增强感知 + 知识库扩容

> 业主令"智能体必须更加智能, 把现有服务(向量库/知识库等)全部运用起来"。本轮: ①点亮此前在本地启动中被关停的向量库(Qdrant)+图库(Neo4j); ②让 agent 的感知层从"一次性表名召回"升级为"多源知识 grounding + 主动血缘"; ③扩容向量知识库(数据标准/命名词根)。

## 一、深度调研结论
- **关键发现**: 本地 app 此前以"仅 DB+crypto"env 启动, `VECTOR_ENABLED/GRAPH_ENABLED` 默认 false → **向量库/图库实际是关的**, agent 的 RAG/语义检索/图谱/向量记忆全降级跑 MySQL 兜底。两库远程 38.76.183.50:6333/7474 可达。
- 向量索引(VectorIndexService)已覆盖 表/术语(glossary)/指标(metric); **缺**数据标准(数据元)/词根/列级。
- 语义检索(SemanticSearchService)支持 kind 过滤 + 向量优先降级关键字; RAG 自动 grounding(buildRagText)为循环前一次性召回。

## 二、落地

### A. 全服务点亮(运维)
本地启动补全 env: `VECTOR_ENABLED=true QDRANT_URL=http://38.76.183.50:6333 QDRANT_API_KEY=… GRAPH_ENABLED=true NEO4J_HTTP=http://38.76.183.50:7474 NEO4J_PWD=…`。`/api/ai/store/health` → vector.ready/graph.ready/embedding.available(1024维) **全 true**。

### B. 知识增强感知(AiAgentService + PromptBuilder)
1. **多源 grounding kind 中文化**: buildRagText 原把 glossary/metric 命中也标"表"。改 `kindLabel()` 映射: table→数据表 / glossary→业务术语 / metric→指标 / dataelement→数据标准 / wordroot→词根 / column→字段; glossary 用"：定义"、metric 用" 口径："呈现 —— 业务术语定义与指标口径作为**领域知识**被正确surface, 不再只是表名。RAG 段头改"# 相关知识(语义召回: 数据表/业务术语/指标口径…)"。
2. **主动血缘注入**: buildBizCtxText 中当 bizCtx 有 db+table → 调 `LineageEdgeService.impact` 拉下游表, 注入"该表下游血缘(改动需评估影响): a、b…"。agent 一上来即知联动关系(主动用上图/血缘库), 异常/无边静默 null。
3. **知识运用纪律**(协议第7条): 涉及数据资产/口径/血缘时, 先用已注入知识或调 semantic_search / graph_* 检索核实再下结论, 结论须与证据一致, 不臆测表名/口径。

### C. 知识库扩容(VectorIndexService + semantic_search)
- 新增 `indexDataElement()`(kind=dataelement, 数据标准: 名称/编码/类型/值域/说明) + `indexWordRoot()`(kind=wordroot, 命名词根: 中文=英文/缩写/分类), 接入 fullReindex。
- semantic_search 工具描述更新: 检索范围"数据表/业务术语/指标/数据标准/命名词根", kind 增 dataelement/wordroot。

## 架构师·全盘结论
- 系统**功能极丰富但深链基础设施缺失**：`govGoModule(key)` / `mdmGoModule(key)` / `navigateTo(route)` 三大跳转入口**只收 key/route、不收上下文 ctx** → 39 个"孤岛"本质是"渲染器收不到下钻参数"。**打通三入口 = 一次性盘活 30+ 死胡同。**
- 联动枢纽实体：表元数据 / 质量规则 / 治理工单 / 指标 / 血缘边 / 消费日志 —— 改一处牵动全局，必须双向打通。
- 5 条端到端业务闭环链(质量闭环 / 指标可信度闭环 / 变更影响闭环 / 敏感治理闭环 / 主数据消费闭环)。

## 业务员·价值评估
- 高价值需强化联动：资产/质量规则/工单/指标驾驶舱/血缘/同步任务/黄金记录。
- 合理但孤立需接入：Profiler/僵尸指标/资产影响面/MDM质量监控/系统监控/统计表/消费审计。
- 伪需求建议弱化合并：指标分类空tab/成员权限矩阵静态页/消费排行与热度重复/冗余接口/死代码(后续轮清理)。

## 本轮改动(大版本)

### 基础设施(workspace.html)
- `switchGovModule(el,key,ctx)` / `govGoModule(key,ctx)` → 设 `window.__govCtx`；`switchMdmModule/mdmGoModule` → `window.__mdmCtx`；`navigateTo(route,ctx)` → `window.__navCtx`。
- ROUTES 各 init 消费 ctx：governance→`govGoModule(ctx.gov,ctx)`、mdm→`mdmGoModule(ctx.mdm,ctx)`、catalog→`dmOpenTable(ctx.openTable)`、metrics→暂存 `__metricsCtx`。
- 数据地图表详情头部加**治理按钮组**(血缘/质量/敏感/治理资产)→ `navigateTo('governance',{gov,table})`；资产影响面指标行加「查看指标」→ `navigateTo('governance',{gov:'consumption',focusMetric})`。

### 六大治理模块接入深链(gov-*.js, 6 代理并行)
- **quality**：收 `ctx.prefillRule` 自动弹预填新建规则 / `ctx.table` 列表按库表过滤；规则行加「查看相关工单」→ health。
- **health**：收 `ctx.issueFilter` 工单按关联对象过滤 / `ctx.dim|date` 滚动定位；工单行「查看对象」(qrule→quality / metric→consumption)；薄弱维度「去改进」→ 对应模块。
- **lineage**：收 `ctx.table` 自动以该表为中心画血缘图；图节点双击设为新中心。
- **classification**：收 `ctx.table` 自动聚焦该表敏感分级；识别结果行「查看该表」→ catalog。
- **assets**：收 `ctx.table` 自动开资产详情抽屉 / `ctx.subjectId` 过滤；成本明细行→详情 / 建生命周期策略。
- **consumption**：收 `ctx.focusMetric` 高亮定位指标行 / `ctx.newDataset` 预填新建数据集；僵尸行加「立即计算/编辑」；概览卡/热度/排行可下钻定位。

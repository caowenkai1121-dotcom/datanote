## 一、诊断:read_tool_result 冗余往返是剩余大头
R111 并行多工具后, 复盘发现"查多表字段"仍偶现慢(实测一轮 237s): agent 并行 3 个 asset_detail 后, 又并行 3 个 read_tool_result, 甚至再叠 delegate_task。根因:
- asset_detail 返回 `{detail:{table, columns:[DnColumnMeta×N]}}`, 每个 DnColumnMeta 含 ~10 字段(id/类型/注释/分类/owner/ordinal…)。72 列 → JSON 远超 trace 折叠阈 TRACE_RESULT_CAP=4000 → 被折叠 → agent 看不全字段 → 调 read_tool_result 取全量 → 凭空多出一轮往返。

## 二、改进:字段清单在 trace 里紧凑化
appendTrace 对含字段清单的结果(asset_detail 的 `detail.columns` 或顶层 `columns`)压成紧凑串:
`表元数据+字段清单(N字段, 表=X, 表注释=...): name1(type1)[注释], name2(type2), ...`
- 72 列紧凑后约 1800 字符 < 4000 → 整张放下不折叠 → agent 一次拿全字段 → 【省去 read_tool_result 往返】。
- 仅压 trace(agent 内联所见); 入库 result_data 仍存全量(cap 16000), 需细粒度元数据(分类分级等)仍可 read_tool_result 取。
- compactColumnsIfAny 兼容驼峰/下划线字段名 + firstNonEmpty 兜底 + try/catch 健壮; 紧凑串超 4000 才回退原折叠逻辑。

# 第 15 轮 · 指标-资产影响联动 + 指标消费排行（跨域联动新功能）

> 接消费层(R10-14)。本轮打通 **metadata 资产域 ↔ consumption 消费域** 的联动，并补消费可观测。
> 业务价值：①改一张物理表前，先看「哪些指标在消费它」评估影响面（经 governance `DnMetricRef` 反查）；②按消费日志看「指标消费排行」，识别高价值/冷门指标。
> 红线遵守：纯**读聚合**，复用既有 `DnMetricRef`(治理域血缘)+`dn_consumption_log`(消费审计)+`DnMetric`，不新建表、不改写路径、零迁移。

## 角色

- **【Domain】** 真实场景：数据工程师下线/改表前必须知道下游影响；指标负责人要知道哪些指标真被用、哪些没人看（治理预算分配）。两者此前**断裂**——资产域看不到消费、消费域看不到资产血缘。本轮把两域接通，场景成立。
- **【Architect】** 联动方向：consumption → governance 单向依赖（consumption 引 `DnMetricRef`，反之不引），不形成环。聚合在 service 层（`MetricValueService`），controller 仅转发。无新表、无新依赖，符合领域驱动边界。
- **【Product】** 功能边界：`asset-impact(db,table)` 返回消费该表的指标列表（去重 metricId）+ 各指标最新值/时间 + ref 类型；`metric-ranking` 返回 Top20 指标（按 METRIC_VALUE/HISTORY/CALC 日志计数，带指标名）。
- **【Design】** 复用 `DN.bars` 条形榜 + 既有消费看板卡片栅格，新增「指标消费排行」卡列于「消费热度」之后，视觉一致。
- **【Dev-A】** 后端：`MetricValueService.assetImpact / metricRanking`（注入 `DnMetricRefMapper`）；`ConsumptionController` 加 `/asset-impact`、`/metric-ranking` 两端点。
- **【Dev-B】** 前端：`gov-consumption.js` 加排行卡 + `loadRanking()`；版本 q100r83→q100r84。
- **【QA】** 见下。

## 新增内容

- **后端（domain.consumption，零新表）**
  - `MetricValueService.assetImpact(db, table)`：`DnMetricRefMapper` 按 `db_name+table_name` 查血缘 → 去重 metricId → 拼 `DnMetric` + `latest()` 最新值 → 列表。
  - `MetricValueService.metricRanking()`：`dn_consumption_log` 按 `target_code` 分组计数（target_type∈METRIC_VALUE/HISTORY/CALC）取 Top20 + 回填指标名。
  - `ConsumptionController`：`GET /api/consumption/asset-impact?db=&table=`、`GET /api/consumption/metric-ranking`。
- **前端**：`gov-consumption.js` 新增「指标消费排行（Top20）」卡 + `loadRanking()`（`DN.bars`，label 指标名、value 计数）。版本 **q100r84**。

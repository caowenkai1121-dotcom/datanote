# 第 1 轮 · 治理闭环接点① 质量 → 工单 自动联动

> 路线图 R1。主题：数据质量规则执行失败/异常时，自动生成（并去重刷新）治理工单，打通"发现问题 → 督办整改"闭环（对应 `00-现状调研.md` 断点2）。
> 选型理由：高价值、加法、零表变更、可独立部署，最能体现重构核心原则"功能联动"，且风险最低，适合作为绞杀式重构第一轮。

## 本轮目标
- 质量规则 `failed`（低于阈值）/ `error`（执行异常）时，自动在 `dn_governance_issue` 生成 `issue_type=QUALITY` 工单。
- 同一规则未关闭工单去重（只刷新不重复建单），问题修复关单后再次失败可重新建单。
- 严重度智能判定：执行异常/强阻断规则=HIGH；规则声明优先；否则按通过率推断。
- 工单可在现有「治理健康分 → 工单」列表直接看到来源（标题/对象/描述携带规则与表列）。
- 单元测试覆盖纯逻辑；全量回归绿；部署生产并端到端验证。

## 各角色产出

**【Domain】** 质量→工单是治理闭环最该自动化却缺失的一环（断点2）。判定标准：配一条必失败的规则→执行→**工单自动出现且可流转**，无需人工建单。强调"工单不能重复刷屏"（同一规则反复跑要去重）。

**【Architect】** 收口于唯一执行点 `QualityService.executeRule`（手动 / run-all / 调度三入口都经此），保证全覆盖。工单创建逻辑归 `IssueService`（SRP：质量服务负责"检测"，工单服务负责"决策与落库"）。本轮先用直接调用，后续结构轮迁移到领域事件 `QualityRuleEvaluated`（见 01 方案 2.2）。**红线：工单生成内部兜底，绝不抛异常影响质量执行主流程**（`run-all` 带 @Transactional，建单异常不得回滚质量结果）。零表变更（`dn_governance_issue` 已在生产，`object_ref` VARCHAR(255)/`issue_type` VARCHAR(40) 足够）。

**【Product】** 复用既有工单列表（`/api/gov/health/issues`），不新建页面。`object_ref=qrule:{ruleId}` 作去重键；标题区分"未达标/异常"；描述含 `库.表.列 / 规则类型 / 通过率 / 失败行数 / 来源规则#`。owner 取规则 `created_by`，dimension 取规则维度（缺省"数据质量"）。

**【Design】** 不引入新视觉。质量模块"立即复跑"失败时 toast 追加"，已自动生成治理工单"，给用户即时反馈，与现有 toast 风格一致。

**【Dev-A 后端】**
- `IssueService`：新增 `raiseQualityIssue(rule, run)` + 纯函数 `qualityIssueObjectRef / normalizeSeverity / qualitySeverity / qualityIssueTitle / qualityIssueDescription`；去重查询 `issue_type=QUALITY AND object_ref=? AND status<>CLOSED`，存在则刷新、否则建单；全程 try/catch 兜底。
- `QualityService`：注入 `IssueService`，在 `executeRule` 落库 run 后调用 `raiseQualityIssue`。

**【Dev-B 前端】** `gov-quality.js`：失败复跑 toast 追加工单提示。

**【QA】** 见下「QA 结论」。

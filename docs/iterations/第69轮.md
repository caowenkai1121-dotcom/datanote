# 第69轮 —— 写框架完整性: resume 按已批 args 精确重放(消除 LLM 漂移 + 幂等防重复)

> 修 R68 暴露的已知缺陷: 原 resume 重跑 LLM 主循环 → 模型可能生成与审批时不同的 args(自由文本尤甚)→ 反复审批 / 甚至执行未经批准的参数。本轮把 resume 改为"按审批记录原始 args 精确重放", 给写框架补上完整性保证: **批准的即执行的, 且恰好一次**。

## 一、设计(资料: 现阶段不部署服务器, 仅本地迭代+连远程库 E2E)
- dn_ai_approval 增列 `executed_at`(sql/73, 已建于远程库)。NULL=已批未执行。
- **AiAgentService.resume(sessionId, ctx)** 不再调 run() 重规划, 改为:
  1. 查本会话 status=approved 且 executed_at IS NULL 的写动作, 按 step_seq 顺序;
  2. 逐个: 按 tool 重放——`Validation` 再校验 args → **fail-closed 写前审计 + 回读**(未落库拒执行且不置 executed 以便重试)→ tool.invoke(原始 args) → 结果审计 → 写 SKILL_CALL step → `executed_at` 置位;
  3. 防御: 重放仅限 readOnly=false 且非 DENY(双保险, 审批记录本不该出现禁区/只读);
  4. 末尾一次 LLM 收尾汇报(不再调工具), seedPriorTrace 喂入既往步骤结果使汇报连贯;
  5. executed>0 → 异步 learn 沉淀经验。
- **Controller /resume** 改调 aiAgentService.resume(精简, 不再拼 goal 重跑)。

## 二、关键收益
- **无漂移**: resume 执行的就是审批面板上批准的那串 args, 模型不再二次生成, 杜绝"批一套、跑另一套"。
- **幂等**: executed_at 置位后, 二次 resume 不重复执行(返回"无待执行的已批准写操作")。
- **安全不破**: 仍走 fail-closed 写前审计; 重放绕过 Guardrail 是正确的(审批已发生), 但加 readOnly/DENY 双保险。

## 【QA】结论 —— ✅ 通过(重放 + 幂等真 E2E)
- 全量回归 `mvn -o test` **527 / 0fail / 1skip** → BUILD SUCCESS。
- **E2E 铁证**(端口 8099, create_metric):
  - chat → wait_approval(pending id=14), 审批前指标数=0(写未裸跑);
  - 审批 → **resume#1 = done, create_metric ok, 指标落库(id=8)**, LLM 汇报清晰("指标已创建, ID=8, 编码 replay_idem");
  - **resume#2 = done, 返回『没有待执行的已批准写操作』, 指标数仍=1(幂等不翻倍)**。

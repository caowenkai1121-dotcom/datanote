# DataNote 重构总体方案

> 基于 `00-现状调研.md`。PM-Lead 综合 Architect（目标架构 + 绞杀路线）、Domain（减法 + 接点优先）、Product（保留/合并/废弃 + 优先级）三方独立产出。
> **总策略：绞杀者模式（Strangler Fig）**——新结构与旧代码并存，旧端点 `@Deprecated` + 转发，前端逐域切换，确认零调用后物理删除。**每轮结束系统完整可用、可一键部署、可回滚到上一 tag。**

---

## 一、重构后产品定位

**DataNote = 面向数据团队的「一站式数据开发–运维–资产–治理」单体平台。**
主干闭环：`数据源 → 开发/同步 → 调度运维 → 资产地图 → 质量治理 → 消费 → 反馈`。
MDM 与项目空间为增强能力（按业主决策裁剪伪需求）。

---

## 二、目标架构（四层 + 平台基座）

```
┌────────────────────────────────────────────────────────┐
│ 接入层  统一鉴权·限流·审计·错误码                        │
├────────────────────────────────────────────────────────┤
│ 应用层  面向前端的用例编排（BFF，薄）                    │
├────────────────────────────────────────────────────────┤
│ 领域层  8 个限界上下文（业务规则与聚合，厚）             │
│  D1数据源  D2元数据&资产  D3数据集成  D4数据开发          │
│  D5调度&运行  D6数据治理  D7主数据MDM  D8项目协作         │
├────────────────────────────────────────────────────────┤
│ 基础设施层  MyBatis-Plus/Doris/Debezium/StreamLoad/      │
│            DataX/Claude 适配器，仓储实现                  │
├────────────────────────────────────────────────────────┤
│ 平台基座 P0（横切）  IAM(认证+RBAC)·配置中心·告警·        │
│            审计·AI 网关·首页门户/总览聚合                 │
└────────────────────────────────────────────────────────┘
```

### 2.1 目标模块清单（10→8 领域 + 1 基座，各单一职责）

| # | 模块 | 单一职责 | 吸收现有 |
|---|---|---|---|
| D1 | 数据源 datasource | 登记/测试/库表列**在线探查唯一出口** | Datasource + DataMapService 探查部分 |
| D2 | 元数据&资产 metadata | 采集/目录/画像/血缘/术语/收藏评论 **资产唯一真相源** | Metadata + MetadataCenter + AssetDetail + Lineage |
| D3 | 数据集成 integration | 离线全量/增量 + CDC，统一 Job 模型 + 对账/坏行/断点 | sync/* + DataX(降为 connector，去留见待决策) |
| D4 | 数据开发 develop | SQL/脚本/版本/DDL 生成/AI 辅助 | Script + HiveDdl + AiAssist |
| D5 | 调度&运行 orchestration | **唯一**调度编排 + 运行实例 + DAG + 重试 + 补数 + 基线 | LocalScheduler + SchedulerConfig + Task* + Backfill + Baseline + sync 内部调度 |
| D6 | 数据治理 governance | 质量/分级分类/脱敏/生命周期/健康分/工单/指标 | Quality + Classification + Masking + Lifecycle + Health + Metric |
| D7 | 主数据 MDM | 黄金记录全链路（按业主决策精简/冻结） | 11 Mdm*Controller → ≤4 |
| D8 | 项目协作 project | 项目+成员+资产+发布（协作套件按业主裁剪） | Project + 15 Project*Service → ≤6 |
| P0 | 平台基座 platform | IAM/配置中心/告警/审计/AI/门户聚合 | Auth+Rbac / SystemConfig+AlertConfig+SchedulerConfig / Audit / Overview+Dashboard |

**边界铁律**：在线探查只在 D1；离线元数据只在 D2；执行编排只在 D5；门户聚合在 P0、各域只暴露原子指标。彻底消灭"一个 service 既连库又读目录""两套调度/两套同步"。

### 2.2 联动闭环（领域事件驱动，先用 Spring `ApplicationEvent`，无需中间件）

| 事件 | 生产者 | 消费者 | 消除的耦合 |
|---|---|---|---|
| `MetadataCollected` | D2 采集 | D2 目录刷新 / D6 质量覆盖 / D8 资产候选 | 采集后自动登记（补断点1） |
| `JobRunFinished` | D5 运行实例 | D2 血缘建边 / D6 健康分 / P0 门户 / D3 对账 | 取代各处轮询 run 表 |
| `ScriptVersionCommitted` | D4 | D2 血缘解析 / D5 调度刷新 | 脚本变更自动重建血缘 |
| `QualityRuleEvaluated` | D6 | P0 健康大盘 / **D6 工单生成** | 质量→工单（补断点2） |
| `GoldenRecordPublished` | D7 | D2 资产回写 / 订阅方 | MDM 发布闭环 |
| `AssetMountedToProject` | D8 | D2 资产标记 / D6 治理范围 | 项目-资产-治理打通 |

> 此层是当前系统**最缺**的：现状靠多处重复 SQL 聚合。引入事件后，健康/概览/血缘从"主动拉取重复计算"变为"被动增量更新"——这是消除 4 套概览重复的根因解。

## 三、数据模型整合（走 sql/66+ 幂等迁移，绝不改 01–65）

| 主题 | 现状（碎） | 目标（整合） |
|---|---|---|
| 同步 | DnSyncTask + DnSyncJob | DnSyncJob 单表 + `engine` 字段（DataX 去留见待决策） |
| 运行实例 | SchedulerRun + TaskExecution + Backfill + Baseline | **DnRunInstance 单表** + `runType` + `taskType` |
| 元数据 | TableMeta/ColumnMeta/Comment/Favorite/... | 以 TableMeta/ColumnMeta 为聚合根，加 `datasource_id` 外键 |
| 健康打分 | HealthScore + ProjectHealth 各一套 | 统一维度字典 + 单一明细表 + `scope` 区分 |
| 配置 | system_config + alert_config + scheduler config | 统一 `dn_config(group,key,value,schema)` |
| 用户偏好 | 4 张同构表 | 统一 `dn_user_preference(object_type,object_id)` |
| 工单 | governance_issue + mdm_change_request | 统一工单引擎，`issue_type` 区分 |

## 四、迭代路线图（10 轮，每轮可独立运行 / 可部署 / 可回滚）

> 排序原则（PM-Lead 综合）：**先做高价值、安全、加法的"接点+减法"**（Domain 主张），**再做结构性合并/统一**（Architect），**版本升级垫底**。每轮走固定流程：Domain→Architect→Product→Design→Dev-A/B→QA→PM。

| 轮次 | 主题 | 涉及 | 关键交付 | 验收（可运行可部署） | 风险 |
|---|---|---|---|---|---|
| **R1** | 治理接点① **质量→工单自动联动** | D6 | 质量规则失败 → 自动生成治理工单；前端工单可见来源 | 配规则→跑失败→工单自动出现，端到端；部署回归 | 低（加法） |
| **R2** | 治理接点②③ **脱敏统一编排 + 分类→标签回写** | D6,D4,D3 | 脚本保存/执行、同步执行三处统一脱敏改写（堵安全漏洞）；分类 confirm 回写表标签 | 普通用户跑 SQL 敏感字段被脱敏；分类确认后标签更新 | 中（触执行链路） |
| **R3** | **安全减法 + 配置清场** | 全局 | 下线孤儿 dev.html；`/api/hive` 双路由转发期；凭据/IP 外置；剥离 Mall 示例；补 backfill 建表 | 删除项前全局检索零调用；部署后功能零变化 | 低–中 |
| **R4** | 接入接点① **采集后元数据自动登记** + 元数据三合一 | D1,D2 | DataMapService 拆 探查(D1)/目录(D2)；在线探查归 D1、离线归 D2；Metadata 双 Controller 合并 | 新接数据源→一键采集→数据地图立即可搜；旧端点转发 | 中 |
| **R5** | **调度统一**（最高结构 ROI） | D5 | 6 种上下线→`ScheduleService.online(type,id)`；消除 /api/scheduler 路由冲突；Backfill/Baseline 降 runType；DnRunInstance 统一表(sql/66)；DolphinScheduler 降可选适配器 | 脚本/同步上下线/补数/重试/日志全链路在统一引擎跑通 | 高 |
| **R6** | **同步收敛 + 三合一** | D3,P0 | 统一断点表；sync_job 配置 UI 统一；告警/审计/用户偏好三处合并（sql/67） | 同步全模式 + 断点对账可用；告警/审计/偏好单一入口 | 中–高 |
| **R7** | **MDM/治理收敛 + 伪需求裁剪** | D6,D7,D8 | MDM 11→4 Controller；统一工单引擎（MDM 变更并入）；按业主决策冻结/砍 MDM 全家桶、项目协作 12 表 | 保留能力在合并端点可用；裁剪项前确认无依赖 | 中（依待决策） |
| **R8** | **平台基座归口** | P0 | Auth+Rbac→IAM；配置三合一 `dn_config`；Overview+Dashboard→门户聚合（订阅事件） | 认证/配置/门户单一入口，门户改事件驱动 | 中 |
| **R9** | **前端模块化** | 前端 | 引入构建（Vite/ESM）；workspace.html 按域拆 modules/；事件委托替换 inline onclick；删 dev.html/桩页 | 各域在新结构逐一验证可用，行数大幅下降 | 中–高 |
| **R10** | **技术栈升级 + 闭环收口 + 清旧** | 全栈 | Java 8→17（R10a）→ SpringBoot 2.7→3（R10b，OpenRewrite javax→jakarta）；接通全部领域事件闭环；物理删除 @Deprecated 旧代码 | 全功能在新栈运行；端到端闭环可演示；无死代码 | 高（分子轮可回滚） |

> **执行中调整（PM-Lead，2026-06-07）**：R1 已完成（质量→工单）。R2 经调研测绘后**收敛为"质量恢复→自动关单"**（补全 R1 闭环），原 R2 的"脱敏统一编排"因需先设计 `SELECT *` 列展开 + 用户上下文集中化（且 ETL 写入路径绝不能脱敏）单独成轮、"分类→标签回写"因 `dn_table_meta.tags` 格式未确立单独成轮，二者顺延插入 R3/R4，其后结构性轮次相应后移。详见各轮文档。

**每轮通用门禁（强制，对应团队质量门禁）：** 1) 功能可运行、核心路径无报错；2) Domain 确认符合真实场景；3) 联动已打通非孤岛；4) 无与其它模块重复职责；5) UI 状态齐全（加载/空/错误/边界）；6) QA 缺陷清单全闭环；7) 有可回归验证；8) 一键部署成功、可回滚到上一 tag。

## 五、保留 / 合并 / 废弃（速查）

- **保留（已验证核心）**：脚本开发+版本+AI / 自研同步引擎 / 统一调度+补数+血缘 / 数据地图+采集+指标 / 治理六维 / RBAC+数据源+系统配置。
- **合并**：Metadata 双 Controller→单一；doris/hive 双路由→`/api/warehouse`；告警两套→告警中心；审计三套→审计中心；偏好 4 表→`dn_user_preference`；6 上下线→ScheduleService；MDM 11→4；运行 4 表→DnRunInstance。
- **废弃**：dev.html / governance.html / integration.html 桩 / test-hdfs / dn_script.ds_* / Mall 示例表（生产）。
- **待业主决策**（见 `待我决策.md`）：DataX 体系去留（Domain 保留分工 vs Product 废弃）；MDM 全家桶冻结/砍；项目协作 12 表裁剪；DolphinScheduler 彻底弃用；前端是否上 Vue3；dn_mdm_hierarchy。

## 六、技术选型升级（分阶段解耦，非阻塞）

| 项 | 现状 | 目标 | 时机 |
|---|---|---|---|
| JDK | 1.8 | 17 LTS | R10a（纯编译升级，可回滚） |

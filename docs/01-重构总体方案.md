# DataNote 重构总体方案

> 版本：2026-06-18  
> 目标：用多轮小步迭代，把当前全栈单体收敛为安全边界清晰、模块职责单一、业务链路闭环、UI 统一的中台系统。

## 一、目标架构

【Architect】继续保留单体部署形态，但在代码内收敛为五层：

1. 协议层：Controller 只处理 HTTP 入参、鉴权后的协议适配和响应封装。
2. 应用层：负责跨域编排、事务边界、领域事件发布。
3. 领域层：各模块维护自己的核心规则和状态机。
4. 基础设施层：数据库、DataX、Doris、WebSocket、AI、向量库、图数据库等外部能力统一适配。
5. 前端体验层：以 `docs/设计规范.md`、`dn-design.css`、`dn-common.js` 为唯一 UI 基座。

## 二、模块边界

【Product】模块所有权先按以下口径收敛：

- 元数据：资产目录、库表详情、字段信息、搜索与采集。
- 数据同步：同步任务、执行历史、错误行、DataX 作业，作为数据运维下的子域。
- 数据治理：质量规则、治理问题、分类分级、血缘影响。
- 主数据：实体、黄金记录、匹配合并、变更审批、源表导入。
- 指标管理：指标定义、发布、计算、取值历史。
- 项目管理：项目、成员、任务、资产、发布流。
- 系统管理：用户、角色、权限、配置、审计。
- AI 助手：受发起用户 RBAC 和数据权限约束的统一自动化入口。

## 三、权限与数据流原则

【Domain】权限必须服务真实业务场景：

- 功能权限控制“能不能做动作”。
- 数据权限控制“能不能看这份数据”。
- 项目角色控制“在这个项目里能不能做”。
- 审批权限控制“能不能让关键变更生效”。
- 审计操作者只由服务端当前登录态决定，外部系统身份另设字段记录。

【Architect】高风险数据流统一走应用服务或领域事件：

- 模型发布 -> 资产目录、质量规则、项目任务。
- 质量问题 -> 治理工单 -> 项目任务。
- 主数据发布 -> 订阅方扇出。
- 指标计算 -> 指标快照 -> 消费审计。

## 四、迭代路线图

### 第 1 轮：高风险入口封堵

【Dev-A/B】清理明文配置与默认凭据；认证默认开启；WebSocket 握手收敛；DataX 输出脱敏；敏感 GET 权限补齐；ProcessUtil 超时修复；关键 UI 权限点和登录页设计问题修复。

### 第 2 轮：权限闭环

【Domain】【Dev-A】TABLE ACL 下沉到统一读数入口，覆盖预览、SQL、治理资产、MDM 导入、AI 查询。普通 GET 按模块补 `*:view`，项目 service 层按 ProjectRoles 强制校验。

### 第 3 轮：配置与外部执行安全

【Architect】【Dev-A】统一配置契约；DataX job 绑定任务归属；Shell 任务默认禁用或独立审批；JDBC extraParams 白名单；SQL 错误和日志统一脱敏。

### 第 4 轮：产品与模块去重

【Product】【Architect】同步 IA、导航、权限目录；确定数据同步和数据质量唯一入口；合并重复 Controller/页面；为高频 API 建 DTO 和统一前端 API client。

### 第 5 轮：业务闭环与审批状态机

【Domain】【Product】主数据初次发布纳入审批或明确数据管家直发权限；项目发布拆分提交/审批/上线/回滚权限；模型发布、质量问题、项目任务走应用服务编排。

### 第 6 轮：UI 设计系统收敛

【Design】【Dev-B】逐页迁移旧弹窗、抽屉、z-index、断点、内联 style 到 `DN.*` 组件和 CSS 令牌；建立桌面/移动断点截图回归清单。

## 五、验收门禁

【QA】每轮完成必须满足：

- 核心路径可运行。
- 权限、数据权限和项目角色有回归测试或可复现实验。
- UI 遵守设计系统，至少覆盖空态、错误态、禁用态、加载态。
- 文档、代码、权限目录、导航口径一致。
- 新增或修改的部署配置不包含凭据。

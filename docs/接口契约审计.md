# 前后端接口契约审计(2026-06-16)

工具: api_audit.py 前缀感知交叉比对(前端常拼接URL, 字面量多为前缀)。
规模: 后端端点 451 · 前端 /api 字面量 301。

## 结论一: 前端调用→后端无实现 = 0 处真问题(契约健康)
- 唯一候选 `/api/dbsync` (project.js:1225) 为**假阳**: 是 `projActRouteOf` 里 `path.indexOf('/api/dbsync')` 字符串匹配(审计路径→模块路由), 非真实请求。真实数据同步走 `/api/sync-job`。
- 即: 前端每个 fetch/api 调用都有对应后端实现, 无"前端调死接口"的 bug。

## 结论二: 后端有、前端未引用 = 11 个候选(疑似遗留/运维端点, 删前需后端调用方复核)
| 端点 | 控制器 | 评估 |
|---|---|---|
| GET /api/ai/status | AiAssistController | 前端用 /api/auth/status; 此 AI 状态端点前端未用 |
| GET /api/ai/store/health | StoreController | 向量库健康检查; 可能被探针调用 |
| POST /api/ai/store/sync-columns(+/status) | StoreController | 列级语义索引(R92); 可能被后台/管理触发 |
| POST /api/datax/generate-job | DataxController | 前端用 /api/datax/create-and-sync; 疑旧粒度步骤 |
| POST /api/datax/run | DataxController | 同上, 疑遗留 |
| GET /api/edit-lock/holder | EditLockController | 前端用 acquire(冲突即返holder); 独立 holder 查询未用 |
| POST /api/metadata-center/column/save | MetadataCenterController | 前端字段编辑用 table/save; 单列保存未用 |
| POST /api/metadata-center/columns/batch-save | MetadataCenterController | 前端未用 |
| POST /api/metadata-center/crawl/datasource/{id} | MetadataCenterController | 前端用 crawl/all、crawl/warehouse; 单源采集未用 |
| GET /api/metadata-center/stats | MetadataCenterController | 前端未用 |

注: 后端控制器路径为 类级@RequestMapping + 方法级 组合, 整串不在源码字面量出现, 故不能用整串 grep 判活; 须按"前端类前缀字面量 + 方法段"复核(本报告已做)。删除前还需确认无 cron/外部系统/AI工具(经HTTP)调用。
